Q1 Multiple Choice Questions (1 Mark Each)

Q2 2(a) Batch vs Multiprogramming OS [3 Marks]

Batch Operating System

In a Batch Operating System, users do not interact directly with the computer system. Instead, users prepare their jobs offline (on punch cards or magnetic tape) and submit them to a system operator. The OS groups (or "batches") jobs with similar needs and executes them sequentially without any manual intervention.

  • Advantages: Large volumes of repetitive work can be processed efficiently. It moves the setup overhead away from the CPU.
  • Disadvantages: The major flaw is highly inefficient CPU utilization. Mechanical I/O devices are much slower than the CPU. When a job requests I/O, the CPU remains totally idle waiting for it to finish. There is also no way to prioritize urgent tasks.
  • Example: Early IBM OS/360, payroll processing systems.

Multiprogramming Operating System

Multiprogramming was designed explicitly to fix the idle CPU problem. The OS keeps multiple programs (a subset of all jobs in the job pool) in main memory simultaneously. The core concept is that the CPU is never allowed to be idle as long as there is at least one program ready to run.

  • How it works: When the currently running program needs to wait for an I/O operation (like reading a file), the OS immediately performs a context switch and hands the CPU over to the next program in the Ready Queue.
  • Advantages: Drastically maximizes CPU utilization and throughput. Allows for more complex scheduling.
  • Disadvantages: Requires complex memory management and CPU scheduling algorithms to prevent interference between processes.
  • Example: Windows NT, UNIX systems.

OR 2(a) Time-sharing vs Real-time OS [3 Marks]

Time-sharing (Multitasking) OS

A logical extension of multiprogramming. Time-sharing systems are designed to provide a highly interactive environment for multiple users concurrently. The OS slices the CPU time into tiny intervals (quanta) and rapidly switches between different users' jobs.

  • Key Feature: The switching is so incredibly fast that every user gets the illusion that they are the sole owner of the entire computer.
  • Focus: Minimizing response time so user interfaces feel fluid and interactive.
  • Example: Modern Linux, Windows 11, macOS.

Real-time OS (RTOS)

Unlike general-purpose OSs where delays are annoying, Real-time systems have strict, non-negotiable timing constraints. Processing must be done within a rigorously defined timeframe, or the system fails entirely.

  • Hard Real-time: Missing a deadline results in total system failure or catastrophe (e.g., pacemaker timing, car airbag deployment, missile guidance).
  • Soft Real-time: Missing a deadline degrades performance but doesn't destroy the system (e.g., live video streaming, stock market quoting).
  • Example: VxWorks, FreeRTOS, QNX.

2(b) Binary Semaphore for 1 Printer [4 Marks]

A binary semaphore is a synchronization primitive that acts essentially like a mutually exclusive lock (mutex). It can only hold the values 0 and 1. To solve the problem of 4 processes sharing exactly 1 printer, we must guarantee that only one process can access the printer hardware at any given moment.

// Initialization: 1 means the printer is currently available
semaphore mutex = 1;

// Structure for all 4 Processes (P1, P2, P3, P4):
do {
    wait(mutex); // Entry section: Decrements mutex to 0. If it was already 0, process blocks.

    /* --- CRITICAL SECTION --- */
    // Process is granted exclusive access.
    print_document();

    signal(mutex); // Exit section: Increments mutex to 1. OS wakes up one blocked process.

    /* --- REMAINDER SECTION --- */
    // Continue with non-printing tasks
} while(true);

How it works: When Process A calls `wait()`, the semaphore drops to 0 and Process A prints. If Process B calls `wait()` while A is printing, B sees the semaphore is 0 and is put to sleep in a queue. Once A finishes, it calls `signal()`, waking B up and allowing B to take its turn.

OR 2(b) Peterson’s Flags Initialization [4 Marks]

Peterson’s algorithm is a classic, purely software-based solution to the critical section problem specifically tailored for exactly two processes (Process 0 and Process 1). It brilliantly combines two variables to ensure mutual exclusion, progress, and bounded waiting.

// Initialization variables:
// flag array shows if a process WANTS to enter. 0 = false, 1 = true.
int flag[2] = {0, 0};
// turn dictates WHOSE turn it actually is to enter, acting as a tie-breaker.
int turn;

// Structure for Process 'i' (where 'j' is the opposing process):
flag[i] = 1; // Step 1: I declare that I want to enter
turn = j; // Step 2: I am polite and offer the turn to the OTHER process

// Step 3: Busy wait (spin) IF the other process wants in AND it is their turn.
while (flag[j] == 1 && turn == j);

/* --- CRITICAL SECTION --- */

flag[i] = 0; // Exit: I am done, I withdraw my request to enter

Why it works perfectly: If both processes try to enter simultaneously, both set their flags to 1. However, they will both overwrite the `turn` variable sequentially. The last process to write to `turn` will be the one forced to wait in the `while` loop, successfully breaking the tie and guaranteeing Mutual Exclusion.

Q3 3(a) RAID Key Features [4 Marks]

RAID (Redundant Array of Independent Disks) uses multiple hard drives to provide data redundancy, performance improvements, or both.

RAID 0 (Data Striping)

Files are broken into blocks (stripes) and written across multiple disks simultaneously. This effectively multiplies read/write speeds by the number of disks used.

  • Redundancy: Zero. If even a single disk out of a 4-disk array fails, 100% of the data across all disks is permanently lost.
  • Use Case: Best used for high-bandwidth tasks where data loss is acceptable (like temporary video rendering caches).

RAID 1 (Data Mirroring)

Every byte of data written to the primary disk is exactly duplicated (mirrored) onto a secondary disk in real-time.

  • Redundancy: Incredible fault tolerance. If one drive completely dies, the system continues running seamlessly off the mirror.
  • Drawback: The major downside is cost: storage efficiency is exactly 50%. You must buy 2TB of drives to get 1TB of usable space.
  • Use Case: Ideal for critical OS drives and accounting databases.

RAID 5 (Striping with Distributed Parity)

The most popular enterprise compromise. Data is striped across disks (like RAID 0 for speed), but a mathematical checksum (parity block) is also calculated and distributed across all the disks.

  • Redundancy: Requires a minimum of 3 disks. It can survive the complete failure of exactly 1 disk; the missing data is computationally rebuilt using the surviving data and the parity data.
  • Efficiency: Storage efficiency is highly favorable: (N-1)/N.

OR 3(a) Disk Scheduling Algorithms [4 Marks]

The OS must decide the order to execute read/write requests to minimize the mechanical movement (seek time) of the hard drive's arm.

FCFS (First-Come, First-Served)

The simplest algorithm. Requests are serviced strictly in the order they entered the queue. While perfectly fair (preventing starvation entirely), it is notoriously slow. If requests bounce between track 10 and track 900, the disk head will wildly thrash back and forth, maximizing mechanical delay.

SSTF (Shortest Seek Time First)

The OS calculates the physical distance from the current head position to all pending requests and selects the one closest to it. This vastly improves throughput and minimizes average seek time. However, it suffers from a fatal flaw: Starvation. If a heavy load of requests keeps arriving near the current track, a request for a distant track may be ignored indefinitely.

SCAN (The Elevator Algorithm)

The disk arm begins at one end of the disk and moves relentlessly toward the other end, servicing every request it passes along its path. When it hits the physical edge, it reverses direction and sweeps back. It provides excellent, predictable throughput under heavy loads and completely eliminates the starvation issues present in SSTF.

3(b) Operating System Structures [4 Marks]

The internal architectural design of the Operating System kernel determines how services communicate.

1. Monolithic Structure

Every single OS service (process scheduling, memory management, file systems, device drivers) is compiled into a single, massive executable file running in a highly privileged kernel space.

Pros: Extremely fast execution since services communicate via direct function calls.
Cons: A bug in a trivial component (like an audio driver) can crash the entire operating system (e.g., early UNIX, Linux).

2. Layered Structure

The OS is strictly divided into hierarchical layers (Layer 0 is hardware, Layer N is the UI). A layer can only request services from the layer immediately below it.

Pros: Exceptionally easy to debug systematically from the bottom up.
Cons: Poor performance due to the overhead of passing parameters down through multiple boundaries.

3. Microkernel Structure

Strips the kernel down to the absolute bare essentials (basic IPC, CPU scheduling, minimal memory management). Everything else (file systems, networking) runs as standard user-space background processes (servers).

Pros: Incredible stability and security; if a file system crashes, it can just be restarted without rebooting the PC.
Cons: Noticeable performance drop due to constant message-passing overhead between user space and kernel space (e.g., Mach, QNX).

4. Modular / Hybrid Kernel

The modern standard. It uses a core monolithic kernel for speed, but supports dynamically loadable kernel modules. Drivers can be loaded or unloaded from memory on the fly without recompiling or rebooting the system. Combines the speed of monolithic with the flexibility of microkernels (e.g., Modern Linux, macOS, Windows NT).

OR 3(b) Linux System Calls [4 Marks]

System calls are the API by which a user program requests privileged services from the OS kernel. Let's analyze the specific commands requested:

  • 1. ps (Process Status)

    Note: This is actually a command-line utility program, not a system call. It works by reading the virtual /proc filesystem to gather and report real-time statistics on currently active processes running on the machine.

  • 2. fork()

    The fundamental mechanism for process creation in UNIX/Linux. It creates a new child process by perfectly cloning the parent's memory image, variables, and open files. It is unique because it returns twice: it returns the value 0 to the newly created child process, and it returns the child's new PID to the parent process.

  • 3. wait()

    When a parent process calls wait(), it voluntarily suspends its execution until its child process terminates. This is critical for preventing "zombie" processes, as it allows the parent to read the child's exit status and allows the OS to completely delete the child's Process Control Block from memory.

  • 4. exec() family

    Exec (e.g., execl, execvp) is usually called by the child immediately after a fork(). It entirely destroys the current process's memory space and overwrites it with a brand new executable program loaded from the disk, starting execution from the new program's main() function.

3(c) First, Best, Worst Fit Allocation [6 Marks]

Contiguous memory allocation algorithms dictate how the OS selects a free block of RAM (a hole) to satisfy a new process's memory request.

Available Memory Holes (in order): 100KB, 500KB, 200KB, 300KB, 600KB
Incoming Process Request: 212KB

First Fit

Logic: Allocates the very first hole in the list that is large enough.

Execution: The OS scans linearly. 100KB is too small. 500KB is large enough. The process is loaded there.

Result: Leaves a leftover fragment of 288KB (500 - 212). It is fast but can cause fragmentation early in memory.

Best Fit

Logic: Searches the entire list to find the hole that fits the process perfectly with the absolute minimum leftover space.

Execution: Valid holes are 500, 300, and 600. The tightest fit is the 300KB hole.

Result: Leaves a leftover fragment of 88KB (300 - 212). It is the most efficient use of immediate space, but tends to generate tiny, unusable fragments.

Worst Fit

Logic: Searches the entire list to deliberately allocate the process into the largest available hole.

Execution: The largest hole in the system is 600KB. The process is placed there.

Result: Leaves a massive leftover fragment of 388KB (600 - 212). The theory is that this large leftover block will still be big enough to hold future, moderately sized processes.

Conclusion

For this specific scenario, Best Fit is mathematically the most efficient use of memory for this single allocation, as it wastes the least amount of space (leaving an 88KB gap versus 288KB or 388KB).

OR 3(c) Virtual Memory & Demand Paging [6 Marks]

1. Virtual Memory Abstraction

Virtual Memory is one of the most critical abstractions provided by an OS. It provides an illusion to applications that they possess a massive, contiguous block of RAM, entirely isolating them from the physical reality of the hardware.

Because virtual memory uses a backing store (the hard disk) as an extension of RAM, it allows the OS to run programs whose combined memory requirements heavily exceed the actual physical RAM installed on the motherboard.

2. Demand Paging Implementation

Demand Paging is the specific hardware/software implementation that makes Virtual Memory possible. Instead of loading an entire massive program (like a 5GB video game) into RAM at launch, the OS implements a "lazy loading" strategy.

Pages (typically 4KB chunks of the program) remain sitting on the hard drive. They are loaded into physical RAM only when the CPU explicitly attempts to execute or read from them.

The Page Fault Mechanism

  1. When a program attempts to access a memory address, the CPU checks the hardware Page Table.
  2. If the 'Valid/Invalid' bit is set to Invalid (meaning the page is on the disk, not in RAM), a hardware Page Fault Trap occurs.
  3. The OS kernel immediately takes control and checks if the memory access was legal.
  4. If legal, the OS finds a free frame in physical RAM (using replacement algorithms if necessary).
  5. It schedules a disk read to pull the requested page into the RAM frame.
  6. It updates the Page Table to mark it as Valid, and restarts the instruction that caused the trap seamlessly. The user program never realizes the data wasn't there to begin with.

Q4 4(a) Concurrency Definitions [4 Marks]

1. Race Condition

A catastrophic software bug that arises in multithreaded or multi-process systems when two or more threads attempt to read, modify, and write to shared data at the exact same time.

Because context switches can happen between any CPU instruction, the final value of the data becomes dependent on the unpredictable chronological order of thread execution (non-determinism). For example, two threads simultaneously reading a bank balance of $100 and adding $50 will result in a final balance of $150 instead of $200 because one update overwrites and erases the other.

2. Mutual Exclusion

The foundational principle of concurrency control. It is an absolute guarantee enforced by the system that if Process A is currently executing instructions that modify shared data, absolutely no other process (Process B, C, etc.) is permitted to access that specific data at the same time. They must wait until Process A finishes.

3. Critical Section

The specific, narrow segment of code inside a program where the shared variables, databases, or files are actually being manipulated. Identifying the critical section is key; programmers must wrap synchronization tools (like mutex locks) directly around the entry and exit points of the critical section to enforce mutual exclusion.

OR 4(a) Software vs Hardware Mutual Exclusion [4 Marks]

Mutual exclusion can be implemented purely in software logic code, or it can leverage specialized hardware instructions built directly into the silicon of the CPU. Here is a detailed comparison:

Software Approaches Hardware Approaches
Rely entirely on complex logic algorithms using shared boolean flags and turn variables. Classic examples include Peterson's Algorithm and Dekker's Algorithm. Rely on special, indivisible machine-language instructions baked into the CPU, such as Test-and-Set or Compare-and-Swap (CAS).
Require absolutely zero special support from the CPU architecture. Will run on any basic processor. Requires complex architecture-level support. The memory bus must physically lock during the instruction to prevent parallel cores from interfering.
Notoriously difficult to design correctly for N-processes. Prone to subtle logic bugs and usually restricted to just 2 processes in practical teaching. Much simpler to implement. These atomic instructions form the foundational building blocks used by OS developers to create higher-level tools like Mutexes and Semaphores.

4(b) PCB Structure [4 Marks]

The Process Control Block (often called a Task Control Block) is the central data structure the OS uses to manage programs. When a process is paused during a context switch, all its vital information must be frozen and saved inside the PCB so it can be perfectly restored later. Think of it as the OS's internal passport or ID card for the process.

+---------------------------+
| Process State             | --> Current status (new, ready, running, waiting, terminated)
| Program Counter           | --> Memory address of the very next instruction to execute
| CPU Registers             | --> Saved contents of Accumulators, Index, & Stack pointers
| Memory Management Info    | --> Page table pointers, segment limits, base registers
| Accounting Info           | --> CPU time consumed, Process ID (PID), Owner User ID
| I/O Status Info           | --> List of all open files, allocated network sockets, USBs
| Scheduling Info           | --> Priority level, pointers to scheduling queues
+---------------------------+

OR 4(b) 5-State Transition Diagram [4 Marks]

This diagram represents the complete lifecycle of a process as it is born, moves between RAM and the CPU, interacts with devices, and eventually dies.

        ┌──────────┐
        │   New    │
        └────┬─────┘
             │ admitted (Resources allocated, moved to RAM)
             ▼
        ┌──────────┐
        │  Ready   │◄─────────────┐
        └────┬─────┘              │
             │ scheduler dispatch │
             ▼                    │
        ┌──────────┐              │
        │ Running  │──────────────┘ (timeout/preemption: Time slice expired)
        └────┬─────┘              │
             │ I/O or event wait  │ (e.g. Waiting for user to press a key)
             ▼                    │
        ┌──────────┐              │
        │ Waiting  │──────────────┘ (I/O or event completion: Interrupt fires)
        └────┬─────┘
             │ exit
             ▼
        ┌──────────┐
        │Terminated│
        └──────────┘

Key Transition Mechanics:

  • Ready → Running: The CPU becomes available, and the scheduler selects this process from the Ready Queue and gives it control.
  • Running → Ready (Preemption): A hardware timer interrupt fires. The OS forces the process off the CPU because its fair share of time (quantum) has expired, placing it back in the queue to wait its turn again.
  • Running → Waiting: The process voluntarily yields the CPU because it needs a slow resource (like reading a file from disk) and cannot proceed until the data arrives. It blocks itself to save CPU cycles.
