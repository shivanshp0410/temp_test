Q1 Multiple Choice Questions (1 Mark Each)

Q2 (a) Benefits of Layered OS Structure [3 Marks]

A layered OS divides the system into hierarchical layers where each layer uses services only from the layer below it. This approach hides the complexity of lower layers from the upper layers.

Step-by-Step Architecture:

Layer N: User Interface
Layer 3: File System
Layer 2: Memory Management
Layer 1: Process Scheduling
Layer 0: Hardware

Core Benefits:

  • Modularity — Each layer is designed and tested independently. If the file system (Layer 3) has a bug, only Layer 3 needs fixing. Memory Management (Layer 2) remains entirely unaffected.
  • Ease of Debugging — Debugging is done sequentially from the bottom up. Layer 0 (Hardware) is verified first. Once Layer 0 is flawless, Layer 1 is built and tested, guaranteeing that any new bugs belong *only* to Layer 1.
  • Data Abstraction — Higher layers don't need to know hardware details. They only interact through well-defined APIs.
  • Security and Isolation — A malicious program in the User Interface cannot directly corrupt the Hardware layer because it must pass through all intermediate layers' security checks.
  • Portability — When moving the OS to a new CPU architecture, only Layer 0 (Hardware) requires rewriting. The upper layers remain unchanged.
Major Disadvantage: Performance overhead. A user request to read a file must cascade through every single layer down to the hardware, resulting in multiple function calls and slower execution compared to a monolithic kernel.

OR (a) System Calls: open(), close(), read(), write() [3 Marks]

System calls act as the bridge between a user-level program and the highly protected OS kernel. When a program needs to access a file, it must request the OS to do it on its behalf via these specific calls.

Step-by-Step File I/O Lifecycle:

1. open(pathname, flags, mode)

Purpose: Requests the OS to locate the file and verify permissions.
Step-by-Step: The OS checks if the file exists at `pathname`. It validates the user's rights against the `flags` (e.g., Read-Only). If successful, the OS creates an entry in its internal File Table and returns an integer index called a File Descriptor (fd) to the user program.

2. read(fd, buffer, count)

Purpose: Pulls data from the file into the program's memory.
Step-by-Step: The program passes the `fd` back to the OS, requesting `count` bytes. The OS goes to the disk (or cache), reads the exact bytes, and copies them into the `buffer` array provided by the user. It returns the number of bytes actually read.

3. write(fd, buffer, count)

Purpose: Pushes data from the program's memory to the file.
Step-by-Step: The program provides data in `buffer`. The OS takes this data and schedules it to be written to the disk sector associated with `fd`. It returns the number of bytes successfully written.

4. close(fd)

Purpose: Cleans up resources.
Step-by-Step: The OS forces any remaining unwritten data (buffered data) to the disk. It then deletes the entry from the File Table and invalidates the `fd`, preventing memory leaks and unlocking the file for other programs.

(b) Binary Semaphore for 1 Printer, 4 Processes [4 Marks]

A binary semaphore acts like a single key to a room. It has only two values: 1 (Available) and 0 (Busy). Since there is only 1 printer, we must enforce strict mutual exclusion.

semaphore mutex = 1;  // Initialization: 1 means printer is free

// Structure for all Processes (P1, P2, P3, P4):
do {
    wait(mutex);         // Attempt to acquire the key
    
    /* CRITICAL SECTION */
    // Process has the key. It exclusively uses the printer.
    
    signal(mutex);       // Release the key and wake up anyone waiting
    
    /* REMAINDER SECTION */
    // Do non-printing tasks
} while(true);

OR (b) Counting Semaphore for 3 Tape Drives, 5 Processes [4 Marks]

A counting semaphore controls access to a resource pool with multiple instances. Here, we have 3 tape drives, so the semaphore starts at 3. The value represents the number of available drives. If the value drops below 0, its absolute value tells us how many processes are waiting.

semaphore S = 3;  // Initialization: 3 tape drives available

// Structure for Processes (P1 to P5):
do {
    wait(S);           // P-operation: Decrement S. If S < 0, block process.
    
    /* CRITICAL SECTION */
    // Allocate 1 tape drive and read/write data
    
    signal(S);         // V-operation: Increment S. If S <= 0, wake up a waiting process.
    
    /* REMAINDER SECTION */
} while(true);

Q3 (a) RAID 0, 1, 5 with Key Features [3 Marks]

RAID (Redundant Array of Independent Disks) combines multiple physical disk drives into a single logical unit for data redundancy, performance improvement, or both.

RAID 0 — Striping

Data is broken into blocks ("stripes") and written across multiple disks simultaneously.

  • Redundancy: None. If one disk out of the array fails, all data is permanently lost.
  • Performance: Extremely high. Read/write speeds scale with the number of disks.
  • Use Case: Video editing caching, temporary file storage where speed is critical but data loss is acceptable.

RAID 1 — Mirroring

Every piece of data written to the primary disk is duplicated exactly onto a mirror disk.

  • Redundancy: 100%. The system can survive the loss of one drive seamlessly.
  • Performance: Write speed is equal to a single disk (must write twice). Read speed is faster as data can be pulled from either drive.
  • Use Case: Operating system drives, accounting databases where data integrity is paramount. Storage efficiency is 50%.

RAID 5 — Striping with Distributed Parity

Data and a mathematical "parity" block are striped across all disks. Parity is calculated via XOR logic.

  • Redundancy: Can survive 1 disk failure. Data is rebuilt computationally using the surviving data + parity blocks.
  • Performance: High read speeds. Moderate write speeds due to the overhead of calculating the parity block on every write.
  • Use Case: Enterprise file servers, data warehousing. Offers the best balance. Storage efficiency is (N-1)/N. Minimum 3 disks.

OR (a) Three Disk Scheduling Algorithms [3 Marks]

When multiple processes request data from a hard disk drive (HDD), the OS must decide the order in which to move the mechanical read/write head to minimize mechanical seek time.

  1. FCFS (First Come First Serve): Serviced exactly in arrival order.
    Pros: Fair, no starvation. Cons: Terrible performance. If requests are for track 10, then 900, then 15, the disk arm thrashes wildly back and forth.
  2. SSTF (Shortest Seek Time First): The arm always moves to the requested track that is physically closest to its current position.
    Pros: Minimizes seek time drastically, high throughput. Cons: Can cause starvation. If requests keep arriving near track 50, a request for track 900 may never be serviced.
  3. SCAN (Elevator Algorithm): The disk arm starts at one end of the disk, moves toward the other end servicing all requests in its path, and reverses direction when it hits the physical edge.
    Pros: Solves starvation, provides a predictable upper bound on waiting times, excellent throughput for heavy loads. Behaves exactly like a building elevator.

(b) Types of OS Structures [4 Marks]

The internal architecture of the Operating System kernel.

1. Monolithic Structure

Every OS service (scheduling, file system, drivers) is compiled into one massive executable running in kernel space.

Pros: Blazing fast (services call each other as simple C functions).
Cons: A bug in a minor audio driver can crash the entire system.

2. Layered Approach

OS is broken into distinct layers (Hardware=0, UI=N). Layer N can only communicate with Layer N-1.

Pros: Very easy to test and debug methodically.
Cons: High performance penalty due to multiple API calls passing down layers.

3. Microkernel Structure

Strips the kernel down to bare essentials (IPC, scheduling). File systems and drivers run as user-space programs.

Pros: Extremely secure. If a file system driver crashes, the OS just restarts it.
Cons: IPC (Inter-Process Communication) overhead slows down the system.

4. Modular / Hybrid Kernel

Core kernel is monolithic for speed, but allows dynamically loading/unloading modules (like USB drivers) on the fly without rebooting. Used by Linux and Windows.

OR (b) System Calls and fork() in UNIX [4 Marks]

A system call is the programmatic way in which a computer program requests a service from the kernel of the OS. It requires a context switch from user mode to kernel mode via a software interrupt (trap).

The `fork()` System Call:

In UNIX, `fork()` is the primary method of creating a new process. When `fork()` is called, the OS creates an exact duplicate (clone) of the parent process.

  • The OS allocates a new Process Control Block (PCB) and a new unique Process ID (PID).
  • It copies the parent's code, data, heap, and stack into the child's memory space.
  • Crucial Difference: Both processes resume execution at the exact line immediately following the `fork()` call. They are differentiated only by the return value of the function.
pid_t pid = fork();

if (pid < 0) {
    // OS failed to allocate memory or resources for a new process
    printf("Fork failed");
} else if (pid == 0) {
    // We are inside the CHILD process.
    // The OS returns 0 to the newly created child.
    printf("I am the child");
    exec(...); // The child usually loads a totally different program here
} else {
    // We are inside the PARENT process.
    // The OS returns the PID of the child to the parent.
    printf("I am the parent, my child's PID is %d", pid);
    wait(NULL); // Parent waits for child to terminate to prevent zombies
}

OR (c) Virtual Memory and Demand Paging [6 Marks]

Virtual Memory is an illusion presented by the OS to programs. It allows a process to believe it has a massive, contiguous block of RAM, even if physical RAM is tiny and scattered. The OS uses the hard drive as an extension of RAM (Swap space) to store parts of programs that are not actively being used.

Demand Paging is the mechanism that powers Virtual Memory. It operates on a "lazy" principle: pages (blocks of data) are loaded into RAM only when the CPU actively demands them, rather than loading the entire program at startup.

Step-by-Step Page Fault Handling:

A page fault occurs when the CPU tries to access a virtual address whose corresponding page is currently sitting on the hard disk, not in RAM.

1
Trap to Kernel

The CPU detects the page table valid bit is 0. Hardware triggers an interrupt, handing control to the OS kernel.

2
Validation Check

The OS checks the process's internal table. Was this a valid memory request (just on disk)? Or an illegal memory access (segmentation fault)? If illegal, OS kills the process.

3
Find a Free Frame

The OS looks for empty space in physical RAM. If RAM is full, it invokes a Page Replacement Algorithm (like LRU) to select an old, unused page (victim) to evict.

4
Swap and Load

If the victim page was modified, write it out to disk. Then, schedule a disk read to pull the newly requested page into the free RAM frame.

5
Update Tables & Restart

The OS updates the Page Table (sets valid bit = 1, maps virtual page to new physical frame) and restarts the exact assembly instruction that caused the fault.

Q4 (a) Race Condition, Mutual Exclusion, Critical Section [3 Marks]

1. Race Condition

A software bug that occurs when two or more processes access and manipulate shared data concurrently. The final outcome depends entirely on the unpredictable order (timing) of CPU scheduling.

Execution Trace of a Race Condition (Lost Update Problem):
Shared variable: bank_balance = 100. Both Process A and Process B try to add 50.
1. Process A reads balance (gets 100) into its CPU register.
2. Context Switch occurs!
3. Process B reads balance (gets 100) into its CPU register.
4. Process B adds 50 (register = 150), and writes 150 back to RAM.
5. Process A resumes. It adds 50 to its saved register (100 + 50 = 150), and writes 150 back to RAM.
Result: Balance is 150 instead of the expected 200. $50 vanished because of bad timing.

2. Critical Section

This is the specific segment of code inside a program where the shared variables, files, or databases are being read and modified. In the example above, the code balance = balance + 50; is the Critical Section.

3. Mutual Exclusion

This is the fundamental solution to race conditions. It is a strict rule enforced by the OS: If Process A is currently executing inside its Critical Section, absolutely no other process is allowed to enter their Critical Section that touches the same shared data.

OR (a) Strict Alteration [3 Marks]

Strict Alteration (Turn Variable) is a software-only attempt to solve the critical section problem for exactly two processes (P0 and P1). It relies on a shared variable called turn.

int turn = 0; // Shared variable

// Process P0 logic:
while (turn != 0);  // SPIN AND WAIT (Busy Waiting) if it's not P0's turn
// Execute Critical Section
turn = 1;           // Give the turn to P1

// Process P1 logic:
while (turn != 1);  // SPIN AND WAIT if it's not P1's turn
// Execute Critical Section
turn = 0;           // Give the turn back to P0

Analysis of this Algorithm:

  • ✅ Mutual Exclusion: Achieved. turn can only be 0 or 1. Therefore, only one process can ever bypass the while loop at a time.
  • ❌ Progress: Violated. Strict alternation means P0, P1, P0, P1. If P0 finishes and sets turn=1, but P1 is doing other things and doesn't want to enter the critical section, P0 is permanently blocked from entering again, even though the critical section is completely empty.
  • ❌ Efficiency: Violated due to Busy Waiting (Spin locking). The blocked process sits in an infinite while loop burning 100% of its CPU time quantum doing nothing but checking the variable.

(b) Process Creation and Termination [4 Marks]

Process Creation Steps:

  1. Identification: The OS assigns a unique numeric identifier (Process ID / PID) to the new process.
  2. Resource Allocation: The OS requests free memory and allocates specific segments: the Text segment (executable code), Data segment (global variables), Stack (function calls), and Heap (dynamic memory).
  3. PCB Initialization: The OS creates a Process Control Block (PCB). It initializes CPU registers, program counter (set to the entry point of the program), and state (set to "New").
  4. Linkage: The new process is placed into the OS's Ready Queue, changing its state from "New" to "Ready". It is now waiting for the scheduler.

Process Termination Steps:

  1. Execution Halt: The process executes its final statement and invokes the exit() system call, or the OS kills it due to a fatal error (e.g., dividing by zero, segmentation fault).
  2. Resource Deallocation: The OS immediately reclaims all memory, closes all open files, and frees I/O buffers associated with the process.
  3. Status Update: The process's exit code is saved inside its PCB. The OS sends a signal to the Parent process indicating the child has died.
  4. Zombie/Deletion: The process is technically dead, but its PCB remains in memory (Zombie state) until the Parent reads the exit code using wait(). Once read, the PCB is fully deleted from OS memory tables.

Key Transition Logic (Why processes move):

  • Waiting → Ready: The hard drive finishes reading data. An interrupt fires, waking the process up. It goes back to the Ready Queue to wait for CPU time again.
