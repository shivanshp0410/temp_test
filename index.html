Q5
What is the significance of using placeholder attributes in form inputs with example?
3 Marks

The placeholder attribute provides a short hint that describes the expected value of an input field. It appears inside the field when it is empty and disappears when the user starts typing.

Significance:

  • Provides contextual hints about expected input format
  • Improves user experience and form usability
  • Reduces need for extra instruction text near inputs
  • Works with text, email, password, search, tel, url, textarea
<form>
  <input type="text"     placeholder="Enter your full name">
  <input type="email"    placeholder="example@gmail.com">
  <input type="tel"      placeholder="+91 9876543210">
  <input type="password" placeholder="Minimum 8 characters">
  <textarea              placeholder="Write your message here..."></textarea>
</form>
📌 Note: Placeholder should NOT replace <label> for accessibility. Always use both for screen reader support.

Q9
What is Grail layout in HTML?
8 Marks

The Holy Grail Layout is a classic page layout consisting of a header, footer, a main content area, and two sidebars. Historically hard to achieve with CSS floats, it's now easy with Flexbox or Grid.

💡 Called "Holy Grail" because it was the ideal layout that was elusive to achieve — like the mythical Holy Grail!

Structure Diagram:

+---------------------------------------+
|             HEADER                    |
+----------+---------------+-----------+
|          |               |           |
|  LEFT    |    MAIN       |   RIGHT   |
| SIDEBAR  |   CONTENT     |  SIDEBAR  |
|          |               |           |
+----------+---------------+-----------+
|             FOOTER                    |
+---------------------------------------+

Implementation using Flexbox:

<!DOCTYPE html>
<html>
<head>
<style>
  * { margin: 0; padding: 0; box-sizing: border-box; }
  body {
    display: flex;
    flex-direction: column;
    min-height: 100vh;
  }
  header, footer {
    background: #333; color: white;
    padding: 20px; text-align: center;
  }
  .middle { display: flex; flex: 1; }
  nav    { background: #555; color: white; width: 180px; padding: 20px; }
  main   { flex: 1; padding: 20px; background: #f0f0f0; }
  aside  { background: #777; color: white; width: 180px; padding: 20px; }
</style>
</head>
<body>
  <header>Site Header</header>
  <div class="middle">
    <nav>Left Sidebar</nav>
    <main>Main Content</main>
    <aside>Right Sidebar</aside>
  </div>
  <footer>Site Footer</footer>
</body>
</html>

Implementation using CSS Grid (preferred):

<style>
  body {
    display: grid;
    grid-template-areas:
      "header header header"
      "nav    main   aside"
      "footer footer footer";
    grid-template-columns: 200px 1fr 200px;
    grid-template-rows: auto 1fr auto;
    min-height: 100vh;
  }
  header { grid-area: header; background: #333; color: white; padding: 20px; }
  nav    { grid-area: nav;    background: #555; color: white; padding: 20px; }
  main   { grid-area: main;   padding: 20px; }
  aside  { grid-area: aside;  background: #777; color: white; padding: 20px; }
  footer { grid-area: footer; background: #333; color: white; padding: 20px; }
</style>
📌 Grid method is more readable and powerful — preferred for Holy Grail layouts in modern CSS.

Q12
Explain any five Git commands with syntax and example.
5 Marks

Git is a distributed version control system for tracking changes in code.

CommandSyntaxPurpose
git initgit init [folder]Initialize new repository
git clonegit clone <url>Copy remote repo locally
git addgit add <file> / git add .Stage files for commit
git commitgit commit -m "message"Save staged changes
git pushgit push origin <branch>Upload commits to remote
git pullgit pull origin <branch>Fetch + merge from remote
git statusgit statusShow working tree status
git loggit log --onelineView commit history
git branchgit branch <name>Create/list branches
git mergegit merge <branch>Merge branch into current
# 1. Initialize a repository
$ git init my-project
# Creates .git folder in my-project/

# 2. Clone remote repo
$ git clone https://github.com/user/repo.git

# 3. Stage files
$ git add index.html    # Stage one file
$ git add .             # Stage all changes

# 4. Commit with message
$ git commit -m "Add login page with form validation"

# 5. Push to GitHub
$ git push origin main
# Sends commits to 'main' branch on GitHub
💡 Typical workflow: init → add → commit → push | Or: clone → edit → add → commit → push

Q14
What are branches in Git and why are they used?
3 Marks

A branch is an independent line of development in Git. It allows multiple versions of a project to coexist simultaneously.

         main branch (production)
           |
A---B---C---+----------F
             \        /
              D---E--    ← feature/login branch (merged)

Why Branches are Used:

ReasonExplanation
IsolationNew features/bug fixes don't affect main/production code
Parallel WorkMultiple developers work on different features simultaneously
Safe ExperimentsTry new ideas without breaking production
Code ReviewChanges reviewed via Pull Request before merging
Release ManagementMaintain separate release and hotfix branches
git branch                  # List all branches
git branch feature-login    # Create new branch
git checkout feature-login  # Switch to branch
git checkout -b hotfix      # Create AND switch
git merge feature-login     # Merge into current branch
git branch -d feature-login # Delete merged branch
💡 Common branches: main (production), develop (integration), feature/*, bugfix/*, hotfix/*
Q15
Explain the concept of Pull and Push Requests in GitHub.
4 Marks

Push: The git command that uploads local commits to a remote GitHub repository.

# After making commits locally:
git push origin main
# Sends your commits to 'main' branch on GitHub

Pull Request (PR): A GitHub feature (not a git command) for requesting to merge code from one branch into another. Enables code review and collaboration.

Developer Workflow with Pull Request:

1. Create feature branch
   git checkout -b feature/login

2. Make changes & commit
   git add . && git commit -m "Add login page"

3. Push branch to GitHub
   git push origin feature/login

4. On GitHub.com:
   → Click "Compare & pull request"
   → Add descriptive title & description
   → Assign reviewers
   → Submit Pull Request

5. Reviewer: reviews code, adds comments

6. If approved → Merge PR into main

7. Delete feature branch (cleanup)
PushPull Request
git commandGitHub web feature
Uploads code to remoteRequests code merge
No review processIncludes peer review
Done individuallyTeam collaboration
⚠️ IMP: PRs enable code review, discussion, CI/CD checks, and approval before merging into production.
Q16
Define: Canvas, Figma, Hierarchy, Components, Prototyping
5 Marks
TermDefinition
1. Canvas In HTML5, <canvas> is an element for drawing 2D graphics via JavaScript API. In design tools (Figma/Adobe XD), the canvas is the infinite workspace where UI designs are created and arranged.
2. Figma A cloud-based, collaborative UI/UX design tool used for creating interfaces, prototypes, and design systems. Works in browser — no installation needed. Supports real-time multi-user collaboration, vector editing, and component libraries.
3. Hierarchy Visual hierarchy is the arrangement of design elements to indicate their order of importance. Achieved through: size, color contrast, typography weight, spacing, and position. Guides the user's eye in a logical sequence through content.
4. Components Reusable design or code elements created once and used multiple times. In Figma: a component (e.g., Button, Card) is created as a master and reused as instances. Editing the master component updates all instances automatically. Promotes design consistency.
5. Prototyping Creating an interactive simulation/mockup of a design to test user flows before development. In Figma: frames are connected with interaction arrows to simulate navigation. Types: Low-fidelity (wireframes) and High-fidelity (pixel-perfect). Enables early user testing and stakeholder feedback.
