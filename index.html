Q1
Explain the objectives and functions of operating systems?
▼

Definition

An Operating System is system software that acts as an interface between the user and computer hardware. It manages hardware resources and provides a platform for application programs to run.

Objectives of OS

  • Convenience: Makes the computer easy and comfortable to use
  • Efficiency: Ensures computer resources are used in an efficient manner
  • Ability to Evolve: Should be constructed in such a way that new system functions can be introduced without interfering with existing ones
  • Throughput: Should maximize the number of tasks completed per unit time

Functions of OS

  • Process Management: Creates, schedules, and terminates processes; handles synchronization and inter-process communication
  • Memory Management: Allocates and deallocates memory to processes; manages virtual memory
  • File System Management: Creates, deletes files/directories; manages read/write/access permissions
  • I/O Management: Controls I/O devices through device drivers; manages buffering, caching, spooling
  • Security and Protection: Prevents unauthorized access to system resources
  • Error Detection and Handling: Monitors and responds to hardware and software errors
  • Networking: Manages communication between processes across systems
  • Resource Allocation: Allocates CPU time, memory, I/O devices among competing processes
  • Accounting: Tracks resource usage per user or process
Q2
Explain basic services provided by Operating system on bare Hardware machine?
▼

On a bare hardware machine (without OS), a user cannot do much. The OS provides the following essential services on top of hardware:

  • Program Execution: The OS loads programs into memory and runs them. It handles normal and abnormal termination.
  • I/O Operations: User programs cannot directly interact with I/O devices. The OS provides a safe, uniform interface to hardware (keyboards, disk, printer, etc.)
  • File System Manipulation: Programs need to read/write files, create/delete directories. The OS provides this through a file system.
  • Communication: Processes need to exchange information. The OS supports this through shared memory or message passing.
  • Error Detection: The OS constantly monitors for errors in CPU, memory hardware, I/O devices, and user programs — and takes appropriate action.
  • Resource Allocation: CPU cycles, memory, I/O devices are allocated among multiple users and processes by the OS.
  • Accounting: Keeps track of how much and what kinds of resources each user or job uses.
  • Protection and Security: Ensures all access to system resources is controlled. OS enforces protection (among processes) and security (from outsiders).
Q3
What is System Call in OS? Explain fork() system call in UNIX OS?
▼

System Call

A system call is the programmatic way a user-level program requests a service from the OS kernel. It provides an interface between a running process and the OS. When a system call is made, the CPU switches from user mode to kernel mode via a software interrupt (trap).

Types of System Calls

  • Process Control: fork(), exit(), wait(), exec()
  • File Management: open(), read(), write(), close()
  • Device Management: ioctl(), read(), write()
  • Information: getpid(), time()
  • Communication: pipe(), shmget()

fork() System Call in UNIX

The fork() system call creates a new process called the child process by duplicating the calling (parent) process. Both processes continue execution from the point after the fork() call.

pid_t pid = fork(); if (pid == 0) // This code runs in CHILD process printf("I am child, PID = %d\n", getpid()); else if (pid > 0) // This code runs in PARENT (pid = child's PID) printf("I am parent, child PID = %d\n", pid); else // pid == -1 means error printf("Fork failed!");
  • Returns 0 to the child process
  • Returns child's PID to the parent process
  • Returns -1 on failure
  • Child gets a complete copy of parent's address space (copy-on-write)
  • Both parent and child execute concurrently after fork()

Time-Sharing OS (in detail)

A time-sharing OS allows multiple users to share the computer system simultaneously. The CPU switches between tasks so rapidly that each user feels they have a dedicated machine. This is achieved by dividing CPU time into small units called time slices or quanta.

  • Multiple users can interact with the system at the same time
  • Each process gets a fixed time quantum (e.g., 10ms); if not done, it is preempted and put at the back of the queue
  • Provides quick response time to all users
  • Context switching happens frequently — OS saves state of one process and loads state of another
  • Good for interactive applications
  • Example: UNIX, Linux, Windows
Advantages: Interactive · Fairness · Multiple users Disadvantages: Context switching overhead · Security (multiple users share system)

Distributed Operating System

Characteristics

  • Resource Sharing: Resources (CPU, memory, files) can be shared across nodes
  • Scalability: New nodes can be added without disrupting the system
  • Fault Tolerance: Failure of one node does not crash the entire system
  • Communication: Nodes communicate via message passing over the network

Advantages

  • Increased computational power through parallel processing
  • Reliability and fault tolerance
  • Resource sharing reduces cost
  • Scalable architecture — easy to expand
  • Load balancing across nodes
  • Better data sharing and communication

Examples

Amoeba, LOCUS, Mach, Google's internal distributed systems

Real Time Operating System (RTOS)

  • Strict timing constraints: Processes must complete within defined deadlines
  • Deterministic behavior: Response time is predictable and guaranteed
  • Types:
    • Hard RTOS: Missing deadline = system failure (pacemakers, aircraft control, industrial robots)
    • Soft RTOS: Missing deadline = degraded quality (video streaming, phones)

  • Minimal interrupt latency — fast response to hardware events
  • Small, lightweight kernel
  • Examples: VxWorks, FreeRTOS, QNX, RTLinux

Time-Sharing Operating System

  • Multiple users share the system simultaneously
  • CPU time divided into time slices (quanta) — each user/process gets a quantum
  • Each user feels they have a dedicated machine
  • Interactive — fast response to user input
  • Uses Round Robin scheduling typically
  • Context switching happens frequently between processes
  • Good CPU utilization
  • Examples: UNIX, Linux, Windows

What is an operating System? Explain the abstract view of the components of a computer system.

Operating System

An OS is system software that manages all hardware and software resources of a computer. It provides an environment where other programs can execute. It acts as a resource manager and extended machine (abstracts hardware complexity from users).

Abstract View of Computer System Components

A computer system has four major components:

  • 1. Hardware: Provides basic computing resources — CPU, memory (RAM), I/O devices (keyboard, monitor, disk). This is the physical layer.
  • 2. Operating System: Controls and coordinates use of hardware among various application programs and users. Sits between hardware and applications.
  • 3. Application Programs: Word processors, compilers, databases, games — define the ways in which system resources are used to solve users' computing problems.
  • 4. Users: People, machines, or other computers that interact with the system.
Layered View (bottom to top): ┌─────────────────────────┐ │ Users │ ├─────────────────────────┤ │ Application Programs │ ├─────────────────────────┤ │ Operating System │ ├─────────────────────────┤ │ Hardware │ └─────────────────────────┘

Dual Mode Operation

  • User Mode: Restricted — user programs execute here; cannot directly access hardware
  • Kernel Mode: Privileged — OS executes here; full access to hardware
  • Switching: System call → kernel mode; return from call → user mode

Q9
What is system call? What is interrupt? How it is handled by OS?
▼

System Call

A system call is the mechanism by which a user program requests a service from the OS kernel. It switches the CPU from user mode to kernel mode via a software interrupt (trap). Examples: fork(), read(), write(), open(), exit().

Interrupt

An interrupt is a signal sent to the processor by hardware or software indicating that an event needs immediate attention. It causes the processor to pause its current activities, save its state, and execute a function called the Interrupt Service Routine (ISR).

Types of Interrupts

  • Hardware Interrupt: Generated by I/O devices (keyboard press, disk I/O completion, mouse click)
  • Software Interrupt (Trap/Exception): Generated by programs — division by zero, system call, page fault
  • Timer Interrupt: Generated by the system clock — used for preemptive scheduling
  • Non-maskable Interrupt (NMI): Cannot be disabled; used for critical hardware failures

How OS Handles an Interrupt

  1. Interrupt signal received by CPU
  2. CPU finishes current instruction
  3. CPU saves current state: Program Counter (PC), registers → pushed onto stack (context save)
  4. CPU looks up the Interrupt Vector Table (IVT) to find the address of the ISR for that interrupt number
  5. ISR (Interrupt Service Routine) executes — handles the event
  6. CPU restores the saved state from stack (context restore)
  7. CPU resumes the interrupted program from where it left off
Q10
Give the features of Batch Operating System.
▼

Batch Operating System

In a batch OS, the user submits jobs offline (e.g., on punch cards). An operator groups similar jobs into batches and feeds them to the computer. The OS executes them one by one automatically.

Features

  • No direct user interaction: User submits job and waits for output — cannot interact during execution
  • Jobs grouped in batches: Similar jobs (same language, same requirements) are grouped to reduce setup time
  • High CPU utilization: No idle time waiting for user input — next job starts immediately
  • Job Control Language (JCL): Special language used to describe jobs and their requirements
  • Sequential execution: Jobs typically executed in FIFO order
  • Minimal OS overhead: Simple scheduling, no context switching for user interaction
  • Long turnaround time: User may wait hours to see output
  • Difficult debugging: Errors found only after job completes
Example use cases: Payroll processing, bank statement generation, monthly billing — tasks that are repetitive and don't need real-time interaction.

Q12
Write short notes on following: Real Time Operating System.
▼

Real Time Operating System (RTOS)

An RTOS is an OS designed to serve real-time applications that process data and events within strict time constraints (deadlines). Correctness depends not only on the logical result but also on the time at which results are produced.

Types

  • Hard RTOS: Deadlines are absolute — missing a deadline is considered a system failure. Used in safety-critical systems: aircraft autopilot, pacemakers, nuclear reactor control, anti-lock braking systems.
  • Soft RTOS: Missing a deadline is undesirable but not catastrophic — quality degrades. Used in: video conferencing, audio streaming, smartphones.

Characteristics

  • Deterministic and predictable response times
  • Priority-based preemptive scheduling
  • Minimal interrupt latency
  • Small, efficient kernel
  • Often lacks features like virtual memory, complex file systems

Scheduling in RTOS

  • Rate Monotonic (RM): Static priority — shorter period → higher priority
  • Earliest Deadline First (EDF): Dynamic — process with closest deadline gets CPU

Examples

VxWorks, FreeRTOS, QNX, RTLinux, Windows CE

Q13
Explain Goals of I/O Software.
▼

Goals of I/O Software

  • Device Independence: Programs should be able to access any I/O device without needing to know the specific device. Example: A program that reads from a file should work whether the file is on a hard disk, USB, or CD-ROM without code changes.
  • Uniform Naming: The name of a file or device should not depend on the device type. In UNIX, all devices are named like files (e.g., /dev/sda, /dev/tty) — uniform naming simplifies programming.
  • Error Handling: Errors should be handled as close to the hardware as possible. If a disk controller can correct the error, it should do so. The higher levels should only be told if the error cannot be corrected at a lower level.
  • Synchronous vs Asynchronous I/O:
    • Synchronous (blocking): Process waits until I/O completes
    • Asynchronous (non-blocking): Process continues; notified when I/O completes. OS should support both.
  • Buffering: Data is stored temporarily in a buffer. This handles: (a) speed mismatch between producer and consumer, (b) data arriving in fixed-size blocks when application needs variable-size reads.
  • Sharable vs Dedicated Devices: Some devices (disk) can be shared by multiple processes simultaneously; others (printer, tape) must be used by one process at a time (dedicated). The OS must handle both cases appropriately (spooling for printers).

OS Services

  • Program Execution, I/O Operations, File System Manipulation
  • Communication between processes, Error Detection
  • Resource Allocation, Accounting, Protection and Security

Multiprocessor OS Types

  • Symmetric Multiprocessing (SMP): All processors are equal — each runs its own copy of the OS; they share the same memory. Most modern multi-core CPUs. All processors peer-to-peer. Load is balanced. Example: Linux SMP.
  • Asymmetric Multiprocessing (AMP): One master processor controls the system and assigns tasks to slave processors. Simpler but master can become bottleneck. Used in embedded systems.
  • NUMA (Non-Uniform Memory Access): Each processor has local memory; accessing remote memory is slower. OS must be aware of memory topology for performance.

Define a process. Explain the process state transition with a neat diagram.

Process Definition

A process is a program in execution. It is an active entity that includes: the program code (text section), current activity (program counter, CPU registers), stack (temporary data), data section (global variables), and heap (dynamically allocated memory).

Process State Transition Diagram

┌──────────────────────────────────────────────┐ │ │ admit timeout/ │ preemption ▼ dispatch │ [NEW] ──────▶ [READY] ◀──────────────── [RUNNING] ▲ │ │ │ │ │ I/O complete I/O req │ │ exit() /event occurs │ │ │ ▼ ▼ └────────────────── [WAITING] [TERMINATED]

Transitions Explained

  • New → Ready: Process admitted into system; loaded into memory
  • Ready → Running: CPU scheduler dispatches the process
  • Running → Ready: Process preempted (time quantum expired or higher-priority process arrives)
  • Running → Waiting: Process requests I/O or waits for event
  • Waiting → Ready: I/O completes or event occurs; process moves back to ready queue
  • Running → Terminated: Process finishes execution or is killed

What is thread and what are the differences between user level threads and kernel supported threads? Under what circumstances is one type "better" than the other?

Thread

A thread is the smallest unit of execution within a process. A multithreaded process has multiple threads sharing the same address space (code, data, heap) but each with its own stack, program counter, and register set.

User-Level vs Kernel-Level Threads

User-Level Threads (ULT)Kernel-Level Threads (KLT)
Managed by user-space thread libraryManaged directly by OS kernel
Kernel unaware of individual threadsKernel maintains TCB for each thread
Thread switch: no mode switch → fastThread switch: requires system call → slower
If one thread blocks → ALL threads blockIf one thread blocks → others continue
No true parallelism on multi-coreTrue parallelism on multi-core CPU
Low overhead for creation/managementHigher overhead
Example: POSIX pthreads (user impl)Example: Linux NPTL, Windows threads

When is ULT Better?

  • When thread switching must be extremely fast (real-time, game engines)
  • When threads mostly do computation (no blocking I/O)
  • When running on OS that doesn't support kernel threads
  • When you need thousands of lightweight threads

When is KLT Better?

  • When threads perform blocking I/O (one block shouldn't freeze all threads)
  • When true parallel execution on multi-core is needed
  • When thread priorities need OS enforcement
  • General-purpose applications (web servers, databases)

What Critical Section Problem and list the requirements to solve it. Write Peterson's Solution for the same.
▼

Critical Section Problem

Consider n processes all competing to use some shared data. Each process has a segment of code called the critical section (CS) in which the shared resource is accessed. The critical section problem is to design a protocol that processes can use to cooperate so that only one process is in its critical section at a time.

Structure of each process: entry section ← asks for permission to enter CS CRITICAL SECTION ← shared data accessed here exit section ← signals that CS has been left remainder section

Three Requirements

  • 1. Mutual Exclusion: If process Pi is executing in its critical section, no other process can be executing in their critical sections.
  • 2. Progress: If no process is executing in its CS and some processes wish to enter their CS, only those processes not in their remainder section can participate in deciding who will enter. This selection cannot be postponed indefinitely.
  • 3. Bounded Waiting: After a process has made a request to enter its CS, there is a limit on the number of times other processes are allowed to enter their CS before that request is granted. This prevents starvation.

Peterson's Solution (for 2 processes, Pi and Pj where j = 1-i)

// Shared variables: bool flag[2] = {false, false}; int turn; // Code for Process Pi: flag[i] = true; // Pi wants to enter CS turn = j; // Give chance to Pj first while (flag[j] && turn == j); // Wait if Pj wants AND it's Pj's turn /* ===== CRITICAL SECTION ===== */ flag[i] = false; // Pi is done /* ===== REMAINDER SECTION ===== */

How it satisfies all 3 requirements:

  • Mutual Exclusion: If both set flag=true and both evaluate while: one has turn=j, other has turn=i — only one's while condition is false → only one enters CS.
  • Progress: If Pj doesn't want to enter (flag[j]=false), Pi enters immediately.
  • Bounded Waiting: Pj sets turn=i when it wants to enter next time, so Pi will enter on next attempt.
Peterson's solution is a software-only solution and works for exactly 2 processes. It requires no special hardware instructions.

Monitor

A monitor is a high-level synchronization construct that provides a convenient and effective mechanism for process synchronization. It is an abstract data type that has:

  • A set of programmer-defined operations (procedures) that are mutually exclusive within the monitor
  • Private shared data variables (accessible only from inside the monitor)
  • Condition variables: x.wait() — suspends calling process; x.signal() — resumes one suspended process

Only one process can be active inside the monitor at any instant — mutual exclusion is automatic.

Dining Philosophers Problem

5 philosophers sit at a round table. Each philosopher alternates between thinking and eating. To eat, a philosopher needs both left and right forks (shared with neighbors). Problem: prevent deadlock (all pick left fork simultaneously) and starvation.

monitor DiningPhilosophers { enum {THINKING, HUNGRY, EATING} state[5]; condition self[5]; void pickup(int i) { // philosopher i wants to eat state[i] = HUNGRY; test(i); if (state[i] != EATING) self[i].wait(); // wait if can't eat } void putdown(int i) { // philosopher i finishes eating state[i] = THINKING; test((i + 4) % 5); // check left neighbor test((i + 1) % 5); // check right neighbor } void test(int i) { if (state[(i+4)%5] != EATING // left not eating && state[i] == HUNGRY // i wants to eat && state[(i+1)%5] != EATING) { // right not eating state[i] = EATING; self[i].signal(); // i can eat now } } initialization_code() { for (int i = 0; i < 5; i++) state[i] = THINKING; } } // Philosopher i: DiningPhilosophers.pickup(i); /* EAT */ DiningPhilosophers.putdown(i);
This solution ensures a philosopher can eat only if both neighbors are NOT eating. No deadlock possible. Starvation is possible in extreme cases but deadlock is fully prevented.

// Shared data: int n = BUFFER_SIZE; // buffer capacity Semaphore mutex = 1; // ensures mutual exclusion on buffer Semaphore empty = n; // counts empty buffer slots (initially n) Semaphore full = 0; // counts full buffer slots (initially 0) Item buffer[n]; int in = 0, out = 0; // insertion and removal indices // PRODUCER PROCESS: do { Item item = produce_item(); // generate new item wait(empty); // wait if no empty slots wait(mutex); // get exclusive access buffer[in] = item; in = (in + 1) % n; signal(mutex); // release access signal(full); // signal that one more slot is full } while (true); // CONSUMER PROCESS: do { wait(full); // wait if no full slots wait(mutex); // get exclusive access Item item = buffer[out]; out = (out + 1) % n; signal(mutex); // release access signal(empty); // signal that one more slot is empty consume_item(item); // process the item } while (true);

  • Writer has exclusive access ✅

What is deadlock? List the conditions that lead to deadlock.

Deadlock

A deadlock is a situation where a set of processes are permanently blocked, each waiting for a resource that is held by another process in the same set. None of the processes can ever proceed because each is waiting for an event that can only be caused by another waiting process.

Example: Process P1 holds Resource R1, waiting for R2 Process P2 holds Resource R2, waiting for R1 → Neither can proceed → DEADLOCK

Four Necessary Conditions (Coffman's Conditions)

ALL four must hold simultaneously for a deadlock to occur. Eliminating any one prevents deadlock.

  • 1. Mutual Exclusion: At least one resource must be held in a non-shareable (exclusive) mode. Only one process can use a resource at a time. If another process requests that resource, it must wait.
  • 2. Hold and Wait: A process is currently holding at least one resource and is waiting to acquire additional resources that are currently held by other processes.
  • 3. No Preemption: Resources cannot be forcibly taken away from a process. A resource can only be released voluntarily by the process holding it after it completes its task.
  • 4. Circular Wait: A set of processes {P0, P1, ..., Pn} exists such that P0 is waiting for a resource held by P1, P1 is waiting for P2, ..., and Pn is waiting for a resource held by P0. This forms a circular chain of waiting.
Memory trick: M-H-N-C (Mutual exclusion, Hold-and-wait, No preemption, Circular wait). All four must exist for deadlock — removing even one is sufficient to prevent it.

List Deadlock Recovery Techniques and explain one of them.
▼

Deadlock Recovery Techniques

Once a deadlock is detected, the OS must recover. Two main approaches:

  • 1. Process Termination
  • 2. Resource Preemption

1. Process Termination (explained in detail)

The OS breaks the deadlock by terminating one or more of the deadlocked processes. Two sub-methods:

  • Abort all deadlocked processes: Terminate every process involved in the deadlock. Simple but very costly — all computation done by those processes is lost. They must be restarted from scratch.
  • Abort one process at a time: Terminate processes one by one, running the deadlock detection algorithm after each termination to check if deadlock is resolved. More controlled but overhead of detection after each step.

Victim Selection Criteria

Which process to terminate? Choose the one with minimum "cost":

  • Process with lowest priority
  • Process that has used least CPU time (least work lost)
  • Process with most remaining time to complete (least productive to keep)
  • Process holding fewest resources
  • Process that is interactive (interactive processes are more important to keep)
  • Number of processes that need to be terminated to resolve deadlock

2. Resource Preemption

Forcibly take resources away from some processes (victims) and give them to others to break the deadlock cycle.

  • Select a victim: Choose which process to preempt based on minimum cost
  • Rollback: Preempted process is rolled back to a safe checkpoint (saved state) and must be restarted from that point
  • Starvation risk: If the same process is always selected as victim, it may starve. Solution: include number of times a process has been preempted in the cost factor
Q4
Define: Starvation.
▼

Starvation

Starvation (also called indefinite postponement) is a situation where a process waits indefinitely because other processes continuously receive preference and resources, leaving the starving process permanently unable to proceed — even though no actual deadlock exists.

How Starvation Occurs

  • Priority Scheduling: A low-priority process may never run if high-priority processes keep arriving continuously
  • SJF Scheduling: A long process may starve if short processes keep arriving — it is perpetually pushed back in the queue
  • SSTF Disk Scheduling: A request for a far-away cylinder may starve if requests near the current head keep arriving
  • Resource Allocation: A process needing multiple resources may starve if other processes always get them first

Difference from Deadlock

Process CAN run but keeps getting skippedProcess CANNOT run — blocked waiting for each other
Other processes make progressDeadlocked processes make NO progress
May resolve if workload changesNever resolves without OS intervention

Solution — Aging

Aging is a technique to prevent starvation: gradually increase the priority of a process the longer it waits. Eventually even a low-priority process gains enough priority to run. Example: In Linux, the nice value can be adjusted; some schedulers increase priority of waiting processes over time.

What is Deadlock? List the conditions that lead to deadlock. How Deadlock can be prevented?

Deadlock Definition & Conditions

See Q1 for deadlock definition and four Coffman conditions (Mutual Exclusion, Hold and Wait, No Preemption, Circular Wait).

Deadlock Prevention

Prevention works by ensuring that at least one of the four necessary conditions cannot hold. This is done at system design time.

  • Prevent Mutual Exclusion: Make resources shareable where possible. Example: Read-only files can be shared. However, this is not always possible — some resources (printers, tape drives) are inherently non-shareable.
  • Prevent Hold and Wait:
    • Method 1: Process must request all resources it will ever need at once, before it starts execution. OS grants all or none. Problem: low resource utilization, process may not know all future needs.
    • Method 2: Process must release all currently held resources before requesting new ones. Problem: may not be practical (data in partial state).
  • Prevent No Preemption: If a process holding resources requests another resource that cannot be allocated, all resources it currently holds are preempted (taken back). The process is restarted only when it can get both the old and new resources. Works for resources whose state can be saved/restored (CPU registers, memory). Does NOT work well for printers, tape drives.
  • Prevent Circular Wait: Impose a total ordering on all resource types. Assign a unique number to each resource type. Processes must request resources in strictly increasing order of these numbers. If process needs resource of type r, it must first release all resources with number ≥ r. This mathematically prevents cycles in the resource graph.
Example of Resource Ordering (Circular Wait Prevention): Assign: Tape(1) < Disk(2) < Printer(3) Process must always request in order: Tape before Disk before Printer → No circular wait possible (you can't hold higher-numbered and request lower)

Banker's Algorithm

The Banker's Algorithm is a deadlock avoidance algorithm developed by Edsger Dijkstra. It mimics a banker who decides whether to grant a loan (resource allocation) by checking if the grant keeps the system in a "safe state." If granting the request leaves the system in a safe state, it is approved; otherwise the process must wait.

Key Concept: Safe State

A system is in a safe state if there exists at least one safe sequence of all processes such that each process can be allocated all its maximum needed resources using currently available resources plus resources freed by previously completed processes.

Data Structures (for n processes, m resource types)

StructureSizeMeaning
Available[m]1×mNumber of currently available instances of each resource type
Max[n][m]n×mMaximum demand of each process for each resource type
Allocation[n][m]n×mCurrently allocated resources to each process
Need[n][m]n×mRemaining need = Max − Allocation

Safety Algorithm

1. Work[m] = Available[m] (copy of available) Finish[n] = {false, false, ..., false} 2. Find index i such that: Finish[i] == false AND Need[i] ≤ Work If no such i exists, go to step 4 3. Work = Work + Allocation[i] Finish[i] = true Go to step 2 4. If Finish[i] == true for all i → SAFE STATE Else → UNSAFE STATE (risk of deadlock)

Resource Request Algorithm (for process Pi requesting Request[i])

1. If Request[i] ≤ Need[i] → proceed; else ERROR (exceeded max claim) 2. If Request[i] ≤ Available → proceed; else Pi must wait 3. Tentatively allocate: Available = Available − Request[i] Allocation[i] = Allocation[i] + Request[i] Need[i] = Need[i] − Request[i] 4. Run Safety Algorithm: If SAFE → grant request (make tentative allocation permanent) If UNSAFE → rollback (restore original values), Pi waits

What is RAG? Explain briefly.
▼

Resource Allocation Graph (RAG)

A Resource Allocation Graph is a directed graph used by the OS to describe the current state of resource allocation and detect deadlocks. It provides a visual representation of which processes hold which resources and which processes are waiting for which resources.

RAG Components

  • Vertices (nodes):
    • Process nodes P = {P1, P2, ..., Pn}: Represented as circles
    • Resource nodes R = {R1, R2, ..., Rm}: Represented as rectangles. Dots inside rectangle = number of instances of that resource.
  • Edges (directed arcs):
    • Request edge (Pi → Rj): Process Pi is requesting (waiting for) an instance of resource Rj. Arrow from process circle to resource rectangle.
    • Assignment edge (Rj → Pi): An instance of resource Rj is currently assigned to process Pi. Arrow from resource rectangle to process circle.

Deadlock Detection Using RAG

  • If the graph contains no cycle → no deadlock
  • If the graph contains a cycle AND each resource type has only one instance → deadlock exists
  • If cycle exists but resources have multiple instances → deadlock may or may not exist (need further detection algorithm)
Example RAG showing deadlock: P1 ──request──▶ R1 ──assigned──▶ P2 P2 ──request──▶ R2 ──assigned──▶ P1 Cycle: P1→R1→P2→R2→P1 → DEADLOCK!

All four must hold simultaneously: Mutual Exclusion, Hold and Wait, No Preemption, Circular Wait. (See Q1 for detailed explanation of each.)

  • Avoid starvation by tracking how many times a process has been preempted (include this in victim cost calculation)

Define mutual exclusion. How mutual exclusion can be achieved? Explain.
▼

Mutual Exclusion

Mutual exclusion is the requirement that when one process is executing in its critical section (the code segment that accesses shared resources), no other process is allowed to execute in its critical section at the same time. It ensures serialized access to shared resources, preventing race conditions and data inconsistency.

How to Achieve Mutual Exclusion

1. Disabling Interrupts

CPU interrupts are disabled before entering critical section and re-enabled after. Simple but dangerous on multi-CPU systems (only disables interrupts on one CPU). Not suitable for user programs.

disable_interrupts(); CRITICAL SECTION enable_interrupts();

2. Strict Alternation (Software)

Two processes take turns using a shared turn variable. Enforces mutual exclusion but violates progress (forces alternation even if one process doesn't need CS).

3. Peterson's Solution (Software, 2 processes)

flag[i] = true; turn = j; while(flag[j] && turn == j); // wait CRITICAL SECTION flag[i] = false;

4. Test-and-Set (Hardware Instruction)

Atomic hardware instruction that reads a memory location and sets it to 1 in one indivisible operation. Cannot be interrupted midway.

boolean TestAndSet(boolean *target) { boolean rv = *target; *target = true; return rv; } while(TestAndSet(&lock)); // spin until lock was false CRITICAL SECTION lock = false;

5. Semaphores

Semaphore mutex = 1; wait(mutex); // lock CRITICAL SECTION signal(mutex); // unlock

6. Mutex Locks (Practical)

Binary semaphore used in modern programming. pthread_mutex_lock() and pthread_mutex_unlock() in POSIX. Java's synchronized keyword. Python's threading.Lock().

7. Monitors (High-Level)

Language construct that automatically provides mutual exclusion. Only one process active inside monitor at a time. Java's synchronized methods implement monitors.

Multiprogramming with Fixed Partitions

Memory is divided into a fixed number of partitions at system boot. Each partition has a predetermined size. Each partition holds exactly one process at a time. The degree of multiprogramming equals the number of partitions.

Memory layout example (Fixed): | OS(100K) | Partition1(100K) | Partition2(200K) | Partition3(300K) | Partition4(400K) | If a 50K process loads into Partition2(200K): → 150K is wasted inside the partition = INTERNAL FRAGMENTATION
  • Internal Fragmentation: Space wasted inside a partition when process is smaller than partition
  • Simple to implement — OS just tracks which partitions are free/busy
  • Inflexible — large process cannot fit even if total free memory is sufficient but fragmented
  • Degree of multiprogramming is permanently fixed at boot

Multiprogramming with Variable Partitions

Partitions are created dynamically to exactly fit each process. When a process arrives, the OS finds or creates a hole of exactly the required size. No fixed partition structure.

Memory evolution example (Variable): Initial: | OS | FREE(400K) | Load P1(100K): | OS | P1(100K) | FREE(300K) | Load P2(200K): | OS | P1(100K) | P2(200K) | FREE(100K) | P1 terminates: | OS | FREE(100K) | P2(200K) | FREE(100K) | Load P3(250K): Cannot fit! 100+100=200K free but not contiguous → EXTERNAL FRAGMENTATION
  • No internal fragmentation — partition sized exactly to process
  • External Fragmentation: Many small holes form between partitions over time; total free may be enough but not contiguous
  • Compaction: OS moves all processes to one end to merge all holes into one large block. Very expensive — must stop all processes and physically move data.
  • Allocation strategies: First Fit, Best Fit, Worst Fit

Explain Swapping in Detail.

Swapping

Swapping is a memory management technique where a process is temporarily moved out of main memory (RAM) to a backing store (hard disk / swap partition), freeing RAM for other processes. Later, the process is brought back into memory to continue execution.

Two Operations

  • Swap Out: Move a process from RAM to disk swap area. Frees the RAM for other processes.
  • Swap In: Move the process back from disk to RAM when it needs to run again.

Detailed Steps

  1. OS scheduler decides a process should be swapped out (e.g., blocked/low-priority, RAM needed)
  2. OS saves the entire process memory image to a reserved area on disk (swap space / page file)
  3. PCB is updated — process state changes, memory addresses cleared
  4. The freed RAM is allocated to another process
  5. When the swapped-out process is scheduled to run again (or when memory is free)
  6. OS loads the process image back from disk to RAM (may be at a different physical address)
  7. PCB updated — process placed back in ready queue

Performance Impact

Swapping cost: Process size = 100MB, Disk transfer rate = 100MB/s Swap out time = 100/100 = 1 second Swap in time = 1 second Total per swap = 2 seconds — significant overhead! This is why demand paging (swapping individual pages, not whole processes) is preferred in modern OSes.

Modern Swapping

  • Modern systems use demand paging — only individual pages (4KB) are swapped, not entire processes
  • Linux uses a swap partition (/dev/swap). Windows uses a page file (C:\pagefile.sys)
  • Swapping allows more processes than can fit in RAM → higher degree of multiprogramming

What is memory? Explain the types of memory.
▼

Memory

Memory is a device or electronic component in a computer that stores data and instructions for the CPU to access. It holds the programs being executed and the data they work with.

Types of Memory (Memory Hierarchy)

  • Registers: Fastest memory, located directly inside the CPU. Stores operands being actively processed. Extremely small (few bytes to KB). Access time: <1 nanosecond. Not visible to programmers — managed by the compiler/CPU. Examples: Accumulator, Program Counter, Stack Pointer, Instruction Register.
  • Cache Memory (L1/L2/L3): Very fast SRAM between CPU and RAM. Stores recently/frequently accessed data and instructions. L1 cache: per-core, fastest, 32-64KB. L2 cache: per-core, larger, 256KB-1MB. L3 cache: shared among cores, 8-32MB. Access time: 1-30 ns. Managed automatically by hardware.
  • Main Memory (RAM — Random Access Memory): Primary working memory of the computer. All running programs and their data reside here. Volatile — data lost on power off. Technology: DRAM. Access time: 100ns. Size: typically 4GB-64GB. Directly addressable by CPU.
  • Secondary Storage (HDD/SSD): Persistent non-volatile storage. Programs and files stored here permanently. HDD (Hard Disk Drive): mechanical, 5-10ms access. SSD (Solid State Drive): flash memory, 100µs access. Much slower than RAM but much larger (1TB-10TB) and cheaper per GB.
  • Tertiary / Offline Storage: Used for archiving and backup. Optical disks (CD, DVD, Blu-ray), magnetic tape. Very slow access (seconds to minutes). Largest capacity (PB range), cheapest per GB.
Memory Hierarchy (Fastest → Slowest): Registers → Cache(L1→L2→L3) → RAM → SSD → HDD → Optical → Tape ↑ Fastest, smallest, most expensive per byte ↑ Slowest, largest, cheapest per byte Key principle: Frequently used data kept in fast memory (cache/RAM). Rarely used data kept in slow storage (HDD/SSD).

Memory Hierarchy

Memory hierarchy is the organization of computer memory into multiple levels based on speed, cost, and capacity. Higher levels are faster, smaller, and more expensive. Lower levels are slower, larger, and cheaper. The hierarchy exploits locality of reference — programs tend to access the same data repeatedly (temporal locality) and nearby data (spatial locality).

Memory Compaction

Compaction is a technique to solve external fragmentation in variable partition memory management. The OS shuffles all occupied memory partitions to one end of memory, merging all the free holes into one large contiguous free block.

BEFORE Compaction (external fragmentation): | OS | P1(100K) | HOLE(50K) | P2(200K) | HOLE(80K) | P3(150K) | HOLE(120K) | Total free = 250K but largest contiguous hole = 120K only.   • Requires all processes to be relocatable at execution time (need relocation registers, not absolute addresses)
  • Very expensive: must temporarily halt processes, physically copy data in memory, update all address mappings
  • I/O operations complicates compaction (process with pending I/O cannot be moved)
  • Not used in modern systems — paging eliminates external fragmentation entirely without compaction

Methods to Track and Manage Free Memory

1. Bitmap (Bit Map)

Memory is divided into fixed-size allocation units (e.g., 4KB). A bitmap has one bit per unit: 0 = free, 1 = occupied.

Memory: |unit1|unit2|unit3|unit4|unit5|unit6|unit7|unit8| Status: |busy |busy |free |free |busy |free |free |busy | Bitmap: 1 1 0 0 1 0 0 1
  • Advantage: Simple, compact, always knows exact free memory
  • Disadvantage: Finding N contiguous free units requires scanning — O(n) time. For 4GB RAM with 4KB units: 1 million bits = 128KB for bitmap itself

2. Linked List of Free/Busy Segments

Maintain a linked list where each node represents a memory segment (hole or process): {type, start_address, length, next_pointer}

List: [HOLE|0|100K]→[PROC|100K|200K]→[HOLE|300K|50K]→[PROC|350K|150K]→[HOLE|500K|300K] On free: merge adjacent holes into one larger hole On allocate: split a hole (First/Best/Worst Fit)
  • Advantage: Fast coalescing of freed blocks; no bitmap scan needed
  • Disadvantage: List management overhead; searching for best-fit is O(n)

3. Buddy System

Memory allocated only in powers of 2 (1KB, 2KB, 4KB, 8KB, ...). When a block is freed, the OS looks for its buddy (the adjacent block of the same size) and merges them. Merging continues recursively.

Request 70KB → round up to 128KB (2^7) Split 1MB into: 512KB + 512KB Split first 512KB into: 256KB + 256KB Split first 256KB into: 128KB + 128KB → allocate first 128KB When freed: 128KB + its buddy 128KB → merged to 256KB → merge with buddy → 512KB → ... Fast O(log n) splitting and merging
  • Advantage: Very fast allocation and deallocation; simple free list per power-of-2 size
  • Disadvantage: Internal fragmentation — 65KB request needs 128KB block (63KB wasted)
  • Used in: Linux kernel for physical page allocation (alloc_pages())

4. Slab Allocator

Pre-allocates pools (slabs) of commonly used kernel objects (task_struct, file, inode, socket). Objects are initialized once, then reused — not freed and reallocated each time.

  • Eliminates fragmentation for kernel objects completely
  • Very fast — no need to initialize object fields each time
  • Used in: Linux kernel (kmem_cache_alloc())

Explain different memory allocation algorithms.

Memory Allocation Algorithms for Variable Partitioning

When a process needs memory, the OS must find a suitable free hole. These algorithms decide which hole to use.

1. First Fit

Scan the free list from the beginning and allocate the first hole that is large enough. Stop as soon as a fit is found.

Free holes: [100K] [500K] [200K] [300K] [600K] Request 212K → first fit: 500K (skip 100K, choose 500K) Remaining: [100K] [288K] [200K] [300K] [600K]
  • ✅ Fastest algorithm — stops as soon as it finds a fit
  • ✅ Good utilization in practice
  • ❌ Tends to leave many small fragments at the beginning of memory

2. Best Fit

Search the entire free list and allocate the smallest hole that is large enough. Produces the smallest possible leftover fragment.

Free holes: [100K] [500K] [200K] [300K] [600K] Request 212K → best fit: 300K (smallest hole ≥ 212K) Remaining: [100K] [500K] [200K] [88K] [600K]
  • ✅ Minimizes wasted space within the chosen hole
  • ✅ Better overall memory utilization for varied request sizes
  • ❌ Slowest — must scan entire free list every time
  • ❌ Produces many tiny unusable fragments (too small for most processes)

3. Worst Fit

Allocate from the largest hole available. Theory: leaving the largest possible remnant increases chance it can be used again.

Free holes: [100K] [500K] [200K] [300K] [600K] Request 212K → worst fit: 600K (largest hole) Remaining: [100K] [500K] [200K] [300K] [388K]
  • ✅ Leftover holes are larger — may accommodate future medium-sized requests
  • ❌ Destroys large blocks — prevents fitting large processes later
  • ❌ Performs worst in simulations — worse than both First Fit and Best Fit

4. Next Fit

Like First Fit, but starts searching from where the last allocation was made (not from the beginning). Uses a roving pointer through the free list.

  • ✅ More uniform distribution of allocations throughout memory
  • ✅ Similar speed to First Fit

Protection Domain

A protection domain is a set of <object, rights> pairs. Each pair specifies an object (resource) and the set of operations allowed on it. A process operates within a domain and can only perform those operations permitted in its current domain.

Domain D = { (File1, {read, write}), (File2, {read}), (Printer, {print}), (Memory_segment_A, {read, write, execute}) }

In UNIX, a domain is defined by the pair (UserID, GroupID). Every process runs with a user-id and group-id. When you execute a program with setuid bit set, the process domain switches to the file owner's domain temporarily.

Access Control Matrix

The access control matrix is a conceptual model where rows represent domains (or users), columns represent objects (files, devices), and each cell lists the allowed operations.

Domain / ObjectFile1File2File3Printer
Domain 1 (User: Alice)read, writeread—print
Domain 2 (User: Bob)readread, writeread—
Domain 3 (User: Admin)read, write, executeread, writeread, writeprint

The full matrix is too large to store in practice. It is implemented as either an Access Control List (ACL) or Capability List.

Access Control List (ACL)

An ACL stores the access matrix by columns. Each object has an associated list of (domain, permitted-operations) pairs — i.e., for each object, who can do what.

File1's ACL: [(Alice: read, write), (Bob: read), (Admin: read, write, execute)] File2's ACL: [(Alice: read), (Bob: read, write), (Admin: read, write)] Printer's ACL: [(Alice: print), (Admin: print)]
  • When a process tries to access an object, the OS checks the ACL of that object to verify the process's domain has permission
  • Easy to revoke all access to an object: just delete its ACL entry
  • Hard to find all objects a particular user can access (must scan all ACLs)
  • Used in: UNIX file permissions (rwxrwxrwx), Windows NTFS, Linux extended ACLs (setfacl)

Capability List

A capability list stores the matrix by rows. Each domain has a list of (object, permitted-operations) pairs — what a user/process can do.

Alice's capabilities: [(File1: read,write), (File2: read), (Printer: print)] Bob's capabilities: [(File1: read), (File2: read,write), (File3: read)]
  • Easy to find all objects a user can access
  • Hard to revoke access: must search all capability lists
  • Like a "key" — a process presents its capability to gain access

UNIX Protection Mechanism (practical)

File permissions: rwxr-xr-- Owner: rwx → read, write, execute Group: r-x → read, execute Others: r-- → read only Commands: chmod 754 file.txt → sets owner=rwx, group=r-x, others=r-- chown alice file.txt → changes owner to alice

Create ways for user authentication? Explain each in brief.

User Authentication

Authentication is the process of verifying the identity of a user or entity before granting access to system resources. It answers the question: "Are you who you claim to be?"

Authentication is different from Authorization (which determines what an authenticated user is allowed to do).

Methods of User Authentication

1. Something You Know — Knowledge-Based

  • Password: Most common. User provides a secret string. OS compares against stored (hashed) password. Simple but vulnerable to dictionary attacks, brute-force, shoulder surfing.
  • PIN (Personal Identification Number): Short numeric password. Used in ATMs, phones. Fast but limited key space.
  • Security Questions: Challenge-response (e.g., "What is your mother's maiden name?"). Easy to forget or guess.
  • Passphrase: Longer than password, a sequence of words. More secure and easier to remember.

2. Something You Have — Token-Based

  • Smart Card: Physical card with embedded chip storing credentials. Must be physically present. E.g., employee ID cards, bank cards.
  • OTP (One-Time Password): Time-based or counter-based password valid for a single login session (e.g., Google Authenticator, SMS OTP). Prevents replay attacks.
  • Hardware Token (RSA SecurID): Physical device that generates time-based codes every 30-60 seconds.
  • USB Security Key (FIDO2/WebAuthn): Physical USB/NFC key that cryptographically proves identity. E.g., YubiKey.

3. Something You Are — Biometric-Based

  • Fingerprint Scanner: Scans unique ridge patterns of fingerprint. Fast, convenient. Used in phones, laptops.
  • Facial Recognition: Analyzes facial geometry using camera. Contactless. Used in smartphones (Face ID).
  • Retina/Iris Scan: Analyzes unique patterns of retina or iris. Very high accuracy, used in high-security systems.
  • Voice Recognition: Analyzes vocal characteristics. Used in phone banking.
  • Behavioral Biometrics: Typing rhythm, mouse movement patterns — continuously verifies identity during session.

4. Multi-Factor Authentication (MFA)

Combines two or more of the above methods for stronger security. Example: Password (something you know) + OTP on phone (something you have). Even if one factor is compromised, the account remains protected.

Examples of MFA: - ATM: Card (have) + PIN (know) - Online banking: Password (know) + SMS OTP (have) - Secure facility: Badge (have) + Fingerprint (are) + PIN (know)

Password Security in OS

  • Passwords stored as cryptographic hashes (never plaintext): SHA-256, bcrypt, Argon2
  • Salt added before hashing to prevent rainbow table attacks
  • UNIX stores hashed passwords in /etc/shadow (not world-readable)
  • Password policies: minimum length, complexity requirements, expiry
  • Account lockout after N failed attempts to prevent brute force

Protection Domain

A protection domain defines the scope of operations a process may perform on system resources. It is a collection of <object, access-rights> pairs. A process can only access objects in its current domain and only with the rights specified in that domain.

Domain Switching

A process may need to switch domains during execution. Domain switching is itself a privileged operation controlled by the OS.

  • In UNIX: Each process runs with a User ID (UID) and Group ID (GID) — this defines its domain
  • setuid bit: Allows a program to temporarily run with the permissions of the file's owner (not the calling user). Example: passwd command runs as root to modify /etc/shadow, even when called by a normal user.
  • In rings-based systems: hardware supports multiple protection rings (0 = kernel, 3 = user). Domain switch requires controlled entry points (gates).

Program Threats

Programs can be written to misuse the system environment, causing harm. These are called program threats or malicious code / malware.

Trojan Horse

A Trojan horse (or simply Trojan) is a program that appears useful or legitimate on the surface but contains hidden malicious code that executes when the program runs. Named after the Greek myth — appears to be a gift but contains enemies inside.

How a Trojan Works

  1. Attacker creates a program that looks useful (a free game, a screen saver, a "free antivirus")
  2. Unsuspecting user downloads and installs it
  3. Program performs its advertised function (to appear legitimate)
  4. Secretly, the hidden code executes: steals passwords, creates backdoors, sends user data to attacker, encrypts files (ransomware), or modifies system files
Example Trojan scenario: User downloads "free_game.exe" → Game launches (legitimate function) → Hidden code simultaneously: - Reads user's stored passwords - Sends them to attacker's server - Creates a new admin user for attacker - Disables firewall

Types of Trojan Horses

  • Remote Access Trojan (RAT): Gives attacker remote control of victim's computer
  • Banking Trojan: Steals banking credentials and financial info
  • Rootkit Trojan: Hides malware deep in the OS, hard to detect
  • Ransomware: Encrypts user's files, demands payment to decrypt
  • Dropper: Installs other malware components

Protection against Trojans

  • Download software only from trusted, official sources
  • Use code signing and digital signatures to verify software authenticity
  • Run programs with least privilege — Trojans are limited by the user's permissions
  • Antivirus and behavior-based detection
  • Sandboxing — run untrusted programs in an isolated environment

Trap Door (Backdoor)

A trap door is a secret, undocumented entry point built into a program (usually by the developer). It allows someone who knows about it to bypass normal authentication and security mechanisms. Also called a backdoor.

How Trap Doors are Created

  • Developer includes a hidden login (e.g., a hardcoded username/password) for "maintenance" or debugging during development
  • Intentionally inserted by a malicious insider to allow future unauthorized access
  • Accidentally left in production code after development is complete
Example Trap Door: // Normal login code if (username == input_user && password == hash(input_pass)) { grant_access(); } // HIDDEN BACKDOOR inserted by developer: if (input_user == "admin_dev" && input_pass == "dev_secret_2024") { grant_full_admin_access(); // bypasses all checks! }

Famous Examples

  • Ken Thompson (Unix creator) described inserting a trojan into the C compiler's login program in his 1984 Turing Award lecture — the backdoor would persist even if source code was recompiled
  • NSA allegedly inserted backdoors into encryption standards
  • Juniper Networks router firmware backdoor (2015)

Protection against Trap Doors

  • Code review — thorough inspection of all source code before deployment
  • Open source code review by community
  • Remove all debug/test code before production release
  • Network monitoring — detect unexpected outbound connections
  • Intrusion detection systems (IDS)

Goals of Operating System Security

OS security aims to protect the system's resources, data, and users from threats while allowing legitimate users to work productively. The OS is the first line of defense — it controls all access to hardware and data.

1. Prevention of Unauthorized Access

The OS must ensure that only authorized users and processes can access system resources (files, memory, devices, network). Unauthorized access can lead to data theft, data corruption, or system damage.

Mechanisms: User authentication (login), access control lists (ACLs), protection domains, file permission bits (UNIX: rwx), mandatory access control (MAC) in SELinux.

2. Data Confidentiality

Sensitive user and system data must be protected from unauthorized disclosure. The OS must enforce that processes cannot read memory or files belonging to other processes/users.

Mechanisms: Memory protection (base/limit registers, paging), process isolation (separate address spaces), file encryption, encrypted storage (BitLocker, LUKS).

3. Data Integrity

The OS must ensure that data cannot be modified by unauthorized users or processes. System files (kernel, system libraries) must be protected from tampering by user programs.

Mechanisms: Write-protect bits, read-only file systems, checksums, secure boot (verifies OS integrity before loading), kernel module signing.

4. System Availability

The OS must protect itself and the system from attacks that try to make the system unavailable (Denial of Service). It must ensure fairness in resource allocation so no process can monopolize resources.

Mechanisms: Process quotas (CPU, memory, disk limits), rate limiting, watchdog timers, zombie process cleanup, DDoS protection at network layer.

5. User Authentication and Authorization

OS must verify the identity of users at login and enforce what each authenticated user is allowed to do throughout their session.

Mechanisms: Login (password/biometric), PAM (Pluggable Authentication Modules) in Linux, Windows Active Directory, sudo/runas for controlled privilege escalation.

6. Process Isolation

Each process must be isolated from other processes. One process should not be able to read or write another process's memory, or interfere with its execution.

Mechanisms: Separate virtual address spaces (paging/segmentation), privilege levels (ring 0 kernel, ring 3 user), system call interface as the only controlled entry to kernel.

7. Auditing and Accountability

The OS must log security-relevant events: logins, file accesses, privilege escalations, system changes. This enables detection of intrusions and forensic analysis after a breach.

Mechanisms: System audit logs (/var/log/auth.log, /var/log/syslog), Windows Event Log, auditd daemon in Linux, intrusion detection systems (IDS) like Snort, AIDE (file integrity monitoring).

8. Protection from Malicious Code

The OS must protect itself and user data from viruses, trojans, worms, and other malicious software.

Mechanisms: Secure boot, code signing, sandboxing (AppArmor, SELinux), address space layout randomization (ASLR), data execution prevention (DEP/NX bit), anti-virus integration.

Explain the IPC Problem known as Dining Philosopher Problem.
▼

Dining Philosopher Problem

The Dining Philosopher Problem is a classic synchronization and concurrency problem formulated by Edsger Dijkstra. It models the challenges of allocating shared resources (forks) among competing processes (philosophers) while avoiding deadlock and starvation.

Problem Statement

  • 5 philosophers sit at a circular dining table
  • There is a bowl of rice in the center and 5 chopsticks (forks), one between each pair of adjacent philosophers
  • A philosopher alternates between thinking and eating
  • To eat, a philosopher must pick up both the left AND right chopstick
  • After eating, the philosopher puts both chopsticks down and returns to thinking

Problems it Models

  • Deadlock: If all 5 philosophers simultaneously pick up their left fork and wait for their right fork — circular wait → deadlock! No one can eat.
  • Starvation: Some philosophers may never get both forks if others always take priority.
  • Race Condition: Without synchronization, two philosophers may grab the same fork simultaneously.
Philosopher i: while(true) { THINK pick_up(left_fork[i]); // wait for left fork pick_up(right_fork[(i+1)%5]); // wait for right fork EAT put_down(left_fork[i]); put_down(right_fork[(i+1)%5]); }

Solutions

  • Allow at most 4 philosophers at the table simultaneously — breaks circular wait
  • Asymmetric solution: Odd-numbered philosophers pick left then right; even pick right then left
  • Monitor solution: A philosopher can pick up forks only if both are available (atomic check-and-pickup)
  • Semaphore solution: Use a binary semaphore (mutex) + counting semaphores per fork
The Dining Philosopher Problem is a standard test for concurrency control algorithms. It appears in OS for process synchronization and in databases for transaction management.
Q2
Define: Race Condition, Mutual Exclusion, Throughput.
▼

Race Condition

A race condition occurs when two or more processes or threads access shared data concurrently, and the final result depends on the order/timing of execution. The program's correctness is "racing" against timing — whichever process runs first changes the outcome.

Classic example — two threads incrementing a shared counter: Thread 1: temp1 = counter (reads 5) Thread 2: temp2 = counter (reads 5) Thread 1: counter = temp1 + 1 → counter = 6 Thread 2: counter = temp2 + 1 → counter = 6 ← WRONG! Should be 7 One increment was lost due to race condition.

Solution: Protect the critical section with mutual exclusion (semaphore, mutex, monitor).

Mutual Exclusion

Mutual exclusion is the requirement that when one process is executing in its critical section (accessing shared resource), no other process is allowed to enter its critical section at the same time. Only one process at a time can access the shared resource.

Implementation methods: Semaphores, Mutex locks, Monitors, Test-and-Set hardware instruction, Peterson's solution.

wait(mutex); // lock — enter CS CRITICAL SECTION // only one process here at a time signal(mutex); // unlock — exit CS

Throughput

Throughput is a performance metric that measures the number of processes/jobs completed per unit time. High throughput means the system is completing many tasks efficiently.

  • Measured in: processes/second, transactions/second, jobs/hour
  • In CPU scheduling: SJF gives higher throughput than FCFS for mixed workloads
  • In disk I/O: SSTF and SCAN give higher throughput than FCFS
  • Throughput vs Response Time: Often a trade-off — optimizing for one may hurt the other
  • High throughput = system is being used efficiently, fewer resources wasted

Explain the Problem of Critical Section (CSP) through Producer Consumer Problem. Explain any one Solution in detail.
▼

Critical Section Problem via Producer-Consumer

The Critical Section Problem is best illustrated through the Producer-Consumer problem (also called Bounded Buffer problem):

  • A Producer produces data items and puts them into a shared circular buffer
  • A Consumer takes items from the shared buffer and consumes them
  • The buffer has limited size N (bounded buffer)

Critical Sections in the Problem

  • Producer's CS: Checking if buffer is full, adding to buffer, updating in pointer
  • Consumer's CS: Checking if buffer is empty, removing from buffer, updating out pointer

If both access the buffer simultaneously without synchronization → race condition → data corruption (wrong item count, incorrect buffer state).

Three Requirements for CSP Solution

  • Mutual Exclusion: Only one of producer/consumer in CS at a time
  • Progress: If neither is in CS, one waiting must be allowed to enter
  • Bounded Waiting: No process waits forever

Solution Using Semaphores

Semaphore mutex = 1; // mutual exclusion on buffer (binary) Semaphore empty = N; // count of empty slots (counting) Semaphore full = 0; // count of full slots (counting) int buffer[N]; int in=0, out=0; // PRODUCER: // CONSUMER: do { do { item = produce(); wait(full); // wait if empty wait(empty); // wait if wait(mutex); // enter CS wait(mutex); // enter CS item = buffer[out]; buffer[in]=item; out=(out+1)%N; in=(in+1)%N; signal(mutex); // exit CS signal(mutex); // exit CS signal(empty); // one more empty signal(full); // one full consume(item); } while(true); } while(true);

This solution correctly handles all three CS requirements and prevents both deadlock and data races.

Q4
What is race condition? Explain the producer consumer problem with a fatal race condition.
▼

Race Condition

A race condition exists when the outcome of a program depends on the relative timing or interleaving of multiple concurrent processes/threads accessing shared data. The program "races" to the correct answer but may lose due to unfortunate timing.

Producer-Consumer with Fatal Race Condition

Consider a simple (unsynchronized) producer-consumer with a shared integer counter:

// Shared variable: int counter = 0; // number of items in buffer // PRODUCER (adds item, increments counter): counter++; // This is NOT atomic! It compiles to: // register1 = counter (load) // register1 = register1 + 1 (increment) // counter = register1 (store) // CONSUMER (removes item, decrements counter): counter--; // Also NOT atomic! Compiles to: // register2 = counter // register2 = register2 - 1 // counter = register2

The Fatal Interleaving

Initial: counter = 5 T0: Producer: register1 = counter (register1 = 5) T1: Producer: register1 = reg1 + 1 (register1 = 6) T2: Consumer: register2 = counter (register2 = 5) ← reads OLD value! T3: Consumer: register2 = reg2 - 1 (register2 = 4) T4: Producer: counter = register1 (counter = 6) T5: Consumer: counter = register2 (counter = 4) ← OVERWRITES! Final counter = 4 ← WRONG! Should be 5 One increment was lost — FATAL RACE CONDITION!

Why "Fatal"?

This is fatal because: (1) It leads to incorrect buffer count — producer may overwrite an unconsumed item, or consumer may try to read from an empty buffer. (2) The bug is intermittent and hard to reproduce — only occurs at specific timing. (3) It can cause data corruption, crashes, or security vulnerabilities.

Solution

Use semaphores/mutex to ensure counter++ and counter-- are executed atomically with proper mutual exclusion.

Device Controller

A device controller is a hardware component (chip or circuit board) that directly manages and interfaces with a physical I/O device. It translates general OS commands into device-specific electrical signals.

  • Has local buffer memory, data registers, control registers, and status registers
  • Manages the actual device operation (moving disk arm, reading sectors, sending data packets)
  • Signals the CPU via interrupt when operation is complete
  • Examples: Disk controller (HDD/SSD), USB controller, NIC (network), GPU, audio controller

Device Driver

A device driver is a software module (part of the OS kernel) that provides a uniform software interface to hardware devices, allowing the OS and user programs to communicate with hardware without knowing device-specific details.

  • Written by device manufacturer and loaded into the kernel
  • Provides standard interface: OS calls read()/write() uniformly for any device
  • Initializes and configures the hardware at system startup

Q3

▼

Direct Memory Access (DMA)

DMA is a hardware feature that allows I/O devices to transfer data directly to/from main memory (RAM) without involving the CPU for each byte. The DMA controller handles the data transfer, freeing the CPU to do other work simultaneously.

Why DMA?

Without DMA (Programmed I/O): CPU manually reads each byte from device register and writes to memory. For a 1MB disk read, CPU executes 1 million load/store cycles — very wasteful.

With DMA: CPU sets up the DMA controller once, then continues executing other processes while DMA handles the entire transfer. CPU is interrupted only on completion.

DMA Transfer Steps

  1. CPU initializes DMA controller: sets source address (device buffer), destination address (RAM), byte count, direction (read/write)
  2. CPU issues the I/O command to the device and goes back to executing other processes
  3. DMA controller takes ownership of the memory bus
  4. DMA transfers data between device and memory one bus cycle at a time (cycle stealing)
  5. After each bus cycle, DMA releases the bus briefly so CPU can use it (interleaved)
  6. DMA decrements the byte count after each transfer
  7. When byte count reaches 0: DMA sends an interrupt to the CPU signaling completion
  8. CPU's interrupt handler processes the completion (e.g., marks I/O request as done, wakes up waiting process)

  • Burst Mode: DMA takes bus exclusively until transfer complete — fast but CPU starved
  • Cycle Stealing: DMA and CPU alternate bus cycles — balanced approach (most common)
  • Scatter-Gather DMA: Transfers to/from multiple non-contiguous memory regions
Q4
Explain SSTF and LOOK disk scheduling Algorithms.
▼

SSTF (Shortest Seek Time First)

SSTF selects the pending request that requires the minimum disk arm movement from the current head position — the shortest seek time. It is a greedy algorithm.

Example: Head = 143. Queue: 86, 1470, 913, 1774, 948, 1509, 1022, 1750, 130 Step 1: Distances from 143: 86→57, 1470→1327, 913→770, 1774→1631, 948→805, 1509→1366, 1022→879, 1750→1607, 130→13 Closest = 130 (dist=13) → move to 130 Step 2: From 130: Closest = 86 (dist=44) → move to 86 Step 3: From 86: Closest = 913 → continue... Sequence: 143→130→86→913→948→1022→1470→1509→1750→1774 Total ≈ 1775 cylinders (vs FCFS=7081 — huge improvement!)
  • ✅ Much better throughput than FCFS
  • ✅ Good average response time
  • ❌ Starvation: Requests far from current head position may wait indefinitely if closer requests keep arriving
  • ❌ Not globally optimal (greedy — makes locally best choice but not globally best)

LOOK Algorithm

LOOK is an improvement of SCAN. The arm moves in one direction and services all requests found. Instead of going all the way to cylinder 0 or 4999 like SCAN, LOOK reverses direction at the last actual pending request in each direction.

Example: Head=143, moving UP. Queue: 86,1470,913,1774,948,1509,1022,1750,130 Requests above 143: 913, 948, 1022, 1470, 1509, 1750, 1774 (last = 1774) Requests below 143: 130, 86 (last going down = 86) LOOK sequence: 143→913→948→1022→1470→1509→1750→1774 (REVERSE)→130→86 ↑ stops HERE, not at 4999 Going UP: 143→1774 = 1631 cylinders Going DOWN: 1774→86 = 1688 cylinders Total LOOK = 3319 cylinders vs SCAN Total = 9769 (going all the way to 4999 and back) LOOK saves: 9769 - 3319 = 6450 unnecessary cylinder movements!
  • ✅ More efficient than SCAN — avoids wasteful travel to disk ends
  • ✅ No starvation — all requests eventually served
  • ✅ Better utilization and lower average seek time

C-LOOK (Circular LOOK)

Like LOOK but services in only one direction. After reaching the last request in one direction, jumps directly back to the lowest-cylinder request (not traveling through intermediate empty space). Provides more uniform wait time distribution.

Device Independent I/O Software

This is a layer of the OS I/O subsystem that sits between device drivers (hardware-specific) and user-level I/O programs (hardware-agnostic). Its purpose is to perform I/O functions that are common to all devices, providing a uniform interface regardless of the specific device.

Functions

  • Uniform Interfacing for Device Drivers: Standardizes how device drivers are called, regardless of device type. OS uses device major/minor numbers to look up the correct driver. This allows user programs to use the same system calls (open, read, write, close) for any device.
  • Buffering: Data is held in a buffer area temporarily to handle speed mismatches between device and process, handle data arriving in fixed hardware block sizes differently from what the application needs, and prevent data loss when process isn't ready to receive.
  • Error Reporting: Reports hardware errors to user programs in a uniform, device-independent way. Maps device-specific error codes to standard OS error codes (ERRNO values in UNIX).
  • Allocating and Releasing Dedicated Devices: Manages exclusive access to devices that can only be used by one process at a time (printers, tape drives). Queues requests and grants access one at a time.
  • Device-Independent Block Size: Different disk types have different physical sector sizes. This layer provides a uniform logical block size to higher layers, hiding the physical differences.

Q6

▼

Elevator Algorithm (SCAN)

The Elevator Algorithm (SCAN) gets its name from the way an elevator works — it moves in one direction, stops at each floor (request), then reverses at the end and services floors in the return direction.

How it Works

  1. Disk arm starts at current cylinder, moving in one direction (e.g., increasing)
  2. Services all pending requests in that direction as it passes them
  3. When it reaches the end of the disk (or last request for LOOK variant), reverses direction
  4. Services all pending requests in the reverse direction
  5. Continues back and forth across the disk
Example: Head=53. Queue: 98,183,37,122,14,124,65,67. Moving UP. Going UP: 53→65→67→98→122→124→183→(199 end) Going DOWN: 199→37→14 Sequence: 53→65→67→98→122→124→183→37→14 Total distance: much less than FCFS (640), comparable to SSTF but no starvation

Advantages

  • No starvation — every request will be served within one full sweep
  • Better average seek time than FCFS
  • More predictable response time than SSTF
  • Good throughput for heavy disk loads

Disadvantage

  • Cylinders just passed must wait almost a full disk sweep
  • SCAN goes to disk ends unnecessarily (LOOK fixes this)
  • Not optimal — does not give minimum total seek time

Variants

Q8
Explain the goals of I/O software.
▼

Goals of I/O Software

  • Device Independence: Programs should be able to access any I/O device without knowing the specific device type. A program that reads a file should work whether the file is on an HDD, SSD, USB flash drive, or network drive — without code changes. The OS provides a uniform abstraction.
  • Uniform Naming: The name of a file or device should not depend on the specific hardware device. In UNIX, all devices are named like files (/dev/sda for disk, /dev/tty for terminal). Programs use the same open()/read()/write() calls for all.
  • Error Handling: Errors should be handled as close to the hardware as possible. If a disk controller can correct a read error through retry, it should do so transparently. Only unrecoverable errors should be reported to higher layers.
  • Synchronous and Asynchronous I/O: Support both blocking I/O (process waits until done — simpler to program) and non-blocking I/O (process continues, notified when done — better for performance). Most user programs use synchronous; OS internally uses asynchronous.
  • Buffering: Data is temporarily stored in memory buffers to handle: (a) speed mismatch between device and process, (b) data arriving in hardware-fixed block sizes, (c) preventing process from waiting for every single character.
  • Sharable vs Dedicated Devices: Disks can be shared — multiple processes can use them concurrently (different sectors). Printers are dedicated — OS must queue print jobs and give exclusive access to one at a time (spooling).
Q9
Briefly describe SCAN.
▼

SCAN Disk Scheduling

SCAN (also called the Elevator Algorithm) is a disk scheduling algorithm where the disk arm starts at one end, moves toward the other end, servicing all pending requests it encounters along the way. When it reaches the end (or the last request in that direction for LOOK), it reverses and services requests in the other direction.

Algorithm: 1. Determine current direction of head movement 2. Service all requests in current direction (increasing or decreasing cylinder numbers) 3. When no more requests in current direction, reverse direction 4. Repeat until all requests served

Characteristics

  • Eliminates starvation — every request is served within one full disk sweep
  • Better average seek time than FCFS
  • No extreme cases like SSTF (where far requests starve)
  • Biased toward middle cylinders — they are visited more often than edge cylinders
Example: Head=143 (moving UP). Queue: 86,913,948,1022,1470,1509,1750,1774,130 Sequence: 143→913→948→1022→1470→1509→1750→1774→4999(end)→130→86 Total SCAN = 9769 cylinders LOOK (better): 143→1774(reverse)→86 = 3319 cylinders

Steps for DMA Data Transfer

  1. Initialization: CPU sets up the DMA controller registers:
    • Source Address Register: memory address to read from / device buffer address
    • Destination Address Register: memory address to write to
    • Count Register: number of bytes to transfer
    • Control Register: direction (read/write), mode (burst/cycle-steal)
  2. CPU issues I/O command to the device controller to start the operation (e.g., read from disk sector)
  3. CPU releases the bus and goes back to executing other processes (multitasking)
  4. DMA takes bus control: DMA controller sends a bus request signal. Bus arbiter grants bus to DMA.
  5. DMA transfers data: Moves data directly between device buffer and main memory without CPU involvement. Uses cycle stealing — takes one bus cycle, transfers one word, releases bus.
  6. Count decrement: After each word transferred, Count Register is decremented. Address registers are incremented.
  7. Completion check: When Count Register reaches 0, all data has been transferred.
  8. Interrupt to CPU: DMA controller asserts interrupt request line. CPU receives interrupt after completing current instruction.
  9. CPU handles interrupt: ISR marks the I/O request as complete, wakes up any process waiting for this data, DMA is reset for next use.

inode (Index Node)

An inode is a data structure in the UNIX file system that stores all metadata about a file, except its name and content. Every file and directory in UNIX has exactly one inode. The inode number uniquely identifies a file within a filesystem.

Information Stored in an inode

  • File type (regular file, directory, symbolic link, device, pipe)
  • File permissions (owner/group/others: read/write/execute)
  • Owner UID and Group GID
  • File size in bytes
  • Timestamps: creation time, last modification time, last access time
  • Hard link count (number of directory entries pointing to this inode)
  • Pointers to the actual data blocks on disk (direct, indirect, double-indirect, triple-indirect pointers)
inode does NOT store: - The file name (stored in directory entry) - The file data/content (stored in data blocks pointed to by inode) View inode info: ls -i file.txt → shows inode number stat file.txt → shows all inode details

File Management in UNIX

  • File Types: Regular files (-), Directories (d), Symbolic links (l), Character devices (c), Block devices (b), Pipes (p), Sockets (s)
  • File Permissions: 9 bits: owner(rwx) group(rwx) others(rwx). Example: -rwxr-xr--
  • System Calls: open(), read(), write(), close(), stat(), chmod(), chown(), unlink()

Directory Management in UNIX

A directory is a special file that contains a list of filename-to-inode mappings. UNIX uses a hierarchical (tree) directory structure with root at /.

Linux Kernel

The Linux kernel is the core component of the Linux OS — it is the software bridge between the hardware and user applications. Developed by Linus Torvalds in 1991, it is open-source (GPL license) and forms the base of all Linux distributions (Ubuntu, Fedora, Debian, Android, etc.).

Type

Linux uses a monolithic kernel — all OS services (process management, memory management, device drivers, file systems, networking) run in kernel space (ring 0), providing high performance. It also supports loadable kernel modules (LKM) — device drivers can be loaded/unloaded at runtime without rebooting.

Functions of Linux Kernel

  • Process Management:
    • Creates, schedules, and terminates processes and threads
    • Implements preemptive multitasking with the CFS (Completely Fair Scheduler)
    • Manages process states, PCBs (task_struct in Linux), signals, and IPC
  • Memory Management:
    • Manages physical RAM and virtual memory using paging
    • Implements demand paging and page replacement (swap space)
    • Provides memory protection and isolation between processes
    • Manages the buddy system allocator and slab allocator for kernel memory
  • File System Management:
    • Supports multiple file systems via VFS (Virtual File System) abstraction layer
    • Supported: ext4, XFS, Btrfs, FAT32, NTFS (read), NFS, procfs, tmpfs
    • Manages inodes, directory entries, file descriptors, and buffer cache
  • Device Management:
    • Provides a loadable module system for device drivers
    • Manages character devices (/dev/tty), block devices (/dev/sda), and network devices
    • Uses udev for dynamic device file management
  • Network Management:
    • Implements the TCP/IP networking stack
    • Manages sockets, network interfaces, routing, and firewalling (iptables/nftables)
  • Security:
    • Enforces user/group permissions and capability-based security
    • Supports Linux Security Modules (LSM): SELinux, AppArmor
    • Implements namespaces and cgroups for container isolation (Docker)
  • System Call Interface:
    • Provides ~300+ system calls that user programs use to request kernel services
    • Handles context switch from user mode to kernel mode safely
